AI 帮浏览器搬家:Ladybird 两周搞定 2.5 万行 Rust 迁移

2.5 万行代码从 C++ 迁移到 Rust,传统方式得几个月?Ladybird 浏览器团队用 AI 两周就搞定了,而且零回退. 这不是"AI 自动写代码"的吹牛故事.创始人 Andreas Kling 把话说得很清楚:人主导,AI 干活.他决定迁移哪些模块,用什么顺序,Rust 代码该长什么样,然后通过几百个小 prompt 让 Claude Code 和 Codex 逐步完成翻译. 目标是 LibJS(JavaScript 引擎),包括词法分析器,解析器,AST 和字节码生成器.关键是 Ladybird 已经有 test262 测试套件,每一段生成的代码都能跟 C++ 版本做字节级对比–生成的 AST 和字节码必须完全一致. 结果:2 周搞定,手工得几个月.零回退,所有测试通过. 这模式值得注意:有测试覆盖 + 有可信参照实现 + AI 翻译,比"让 AI 从零写代码"靠谱多了.内存安全语言迁移的老大难问题,可能找到了加速器.

CRTX:让 AI 生成代码后自己跑测试,自己修 bug

刚发现一个有意思的CLI 工具:CRTX ️ 谁能用? AI 生成代码最头疼的是什么?跑不起来.导入报错,测试失败,边界情况没处理.拿到代码还得花 10-30 分钟调试. CRTX 解决这个问题:生成后自动跑测试,失败就自动修复. 用法很简单: pip install crtx crtx loop "Build a REST API with FastAPI, SQLite, search and pagination" 一条命令进去,出来的是已经通过测试的代码. 怎么做到的? CRTX 的核心是 Generate → Test → Fix → Review 循环: Generate - 选合适的模型生成代码 Test - 本地跑 AST 解析,导入检查,pyflakes,pytest Fix - 测试失败?把错误喂回模型让它修 Review - 另一个独立模型(Arbiter)审查最终输出 测试不通过不放行,修复卡住会启动三级 escalation:先诊断根因,再简化上下文重试,最后换个模型给"第二意见". 比别的工具有什么优势? 单模型 Claude, 准确率=94%, 开发者调试时间=10 分钟 单模型 o3, 准确率=81%, 开发者调试时间=4 分钟 多模型辩论, 准确率=88%, 开发者调试时间=9 分钟 CRTX Loop, 准确率=99%, 开发者调试时间=2 分钟 ...

Grok 4.20 Beta炸场:8个AI Agent并行写代码,xAI要把Cursor们逼到墙角

Grok 4.20 Beta在2月17日悄然上线,Reddit的r/singularity瞬间炸了. 但这次让开发者坐不住的不是跑分–而是xAI塞进去的一套多Agent并行编程系统.简单说:你发一个需求,最多8个AI Agent同时帮你写代码,然后系统自动评分,挑最好的方案给你. 这玩意儿,直接把AI编程从"单人作战"推进到了"团队协作"阶段. 8个Agent同时干活,这不是噱头 Grok Build是xAI的编程产品,之前一直在内测.这次随4.20 Beta一起曝光的Parallel Agents功能,是真正的杀手锏. 具体怎么用: 你输入一个编程需求 系统同时启动最多8个Agent(两个模型各4个:Grok Code 1 Fast和Grok 4 Fast) 所有Agent的输出并排显示,你可以逐个对比 界面还带context usage tracker,实时看每个Agent消耗了多少上下文 这不是什么花活儿.Musk之前说过要让Grok"同时调度数百个专业编程Agent协作",现在看来这是第一步落地. 竞技场模式:Agent打擂台 比Parallel Agents更有意思的是代码里发现的Arena Mode. Parallel Agents是让你人工比较多个输出.Arena Mode则更进一步–Agent之间自动竞争,系统自动评分排名,选出最优方案. 这个思路跟Google Gemini Enterprise的Idea Generation Agent类似:先让多个Agent各自生成方案,再通过结构化竞赛流程筛选最佳结果. 区别在于,Google的方案偏向企业级工作流,而xAI直接把这套东西做进了编程工具里.开发者写代码的时候,背后有一场小型锦标赛在运行–这个画面还挺赛博朋克的. 实力到底怎么样? 先说跑分:Reddit社区的反馈是混合的. 有人发现Grok 4.20在SimpleBench测试里会"作弊"–Agent不是纯靠推理,而是手动上网搜答案.这说明xAI在推理能力和工具调用之间还没找到最佳平衡点. 但在实际应用层面,数据很硬: Alpha Arena交易竞赛(2026年1月):Grok 4.20的早期checkpoint拿下了10-12%的回报率,部分配置甚至+47%,而竞争对手全在亏钱, 实时数据处理:接入X平台Firehose(每天约6800万条英文推文),实现1-5分钟级别的情绪-价格信号分析, 多模态能力提升:文本,图像,视频全面增强, 幻觉率降低:内置fact-checking工具 说白了,Grok 4.20的路线不是"在封闭测试里刷高分",而是"在真实世界里能干活". Grok Build:不只是Agent,是一个IDE 除了Agent系统,Grok Build的UI也在大改.现在的界面已经接近一个浏览器端IDE: 语音输入支持(Vibe Coding的核心体验), 导航标签:Edits,Files,Plans,Search,Web Page, 实时代码预览和代码库导航, 分享按钮和评论功能(协作场景), GitHub App集成(设置里已可见,但还没激活) xAI的野心已经很明确了:不是做一个ChatGPT的编程插件,而是做一个原生AI-first的开发环境.Cursor,Windsurf们应该已经感受到压力了. Grok 5:6万亿参数怪兽在路上 Grok 4.20只是过渡.真正的大招是2-4个月后的Grok 5. 关键数据: 参数量, Grok 3/4=3万亿, Grok 5=6万亿 智能密度, Grok 3/4=基准, Grok 5=每GB更高 工具使用, Grok 3/4=基础, Grok 5=大幅增强 ...

AGENTS.md到底有没有用?一篇论文引爆开发者社区大战

如果你最近在用 Claude Code,Codex 或者 OpenClaw 之类的 AI 编程工具,你大概率见过一个叫 AGENTS.md(或 CLAUDE.md)的文件.它的作用很简单:告诉 AI Agent 这个项目的规矩–用什么框架,怎么跑测试,有什么坑要避. 听起来很合理对吧? 然后一篇 arXiv 论文说:这玩意不但没用,还让 Agent 变蠢了. 论文说了啥 这篇来自学术界的论文 “Evaluating AGENTS.md: Are Repository-Level Context Files Helpful for Coding Agents?” 做了一个挺严谨的实验: 在 SWE-bench(业内标准的代码修复基准测试)上跑了多个 Agent + 多个 LLM 的组合 分别测试了三种情况:没有上下文文件,LLM 自动生成的上下文文件,开发者手写的上下文文件, 结果:, LLM 生成的文件:任务成功率平均下降 3% 开发者手写的文件:平均提升 4%,但在不同模型上表现不一致(Sonnet 4.5 反而下降 2%) 所有情况下推理成本增加超过 20% 论文的结论是:上下文文件里的"不必要要求"让任务变难了,建议只写最小化的必要信息. HN 社区炸了 这篇论文在 Hacker News 上拿到 146 分,94 条评论,评论区基本是一边倒的"你们测错了". 最核心的反驳来自一位高赞评论者: “4% 的提升是巨大的!一个简单的 markdown 文件就能带来 4% 的改善,这说明它是必须有的.” 他指出了论文的一个关键盲区:SWE-bench 测试的都是公开的 GitHub 项目,而这些项目里的 AGENTS.md 大多来自"最近的小型 vibecoding 项目",质量参差不齐.真正需要 AGENTS.md 的场景是大型私有代码库–那些有复杂部署流水线,团队约定,历史债务的项目. ...
