Agent Passport:AI Agent 的「身份证」,像 OAuth 一样验证身份

一句话总结 Agent Passport = AI Agent 的身份证–类似 “Sign in with Google”,但专为 Agent 设计.Ed25519 加密验证,JWT 令牌,风险评分,解决 Agent 冒用和安全隐患. 为什么需要它? AI Agent 正在爆发: OpenClaw 18万+ GitHub stars Moltbook 230万 Agent 账户 但问题是:没有标准方式证明 Agent 身份. 这意味着: 恶意 Agent 可以冒充其他 Agent Skill/Plugin 市场没有认证层 Cisco 安全团队已经在第三方 Agent skills 中发现数据泄露 Agent Passport 就是来解决这个问题的. 核心特性 Ed25519 挑战-响应认证 私钥永远不会离开 Agent, 公钥用于验证身份, 抗量子计算攻击 JWT 身份令牌 60 分钟有效期(TTL), 可撤销, 包含 Agent 元数据和权限 ️ 风险评分引擎 0-100 分评分系统: Allow (0-30): 低风险,正常通行, Throttle (31-70): 中等风险,限制频率, Block (71-100): 高风险,拒绝访问 一行代码验证 const result = await passport.verify(token); 快速开始 安装 npm install @zerobase/agent-passport ### Agent 端:生成身份 import { AgentPassport } from '@zerobase/agent-passport'; // 生成密钥对(只需一次) const passport = new AgentPassport(); const identity = await passport.generateIdentity(); // 保存私钥(安全存储) const privateKey = identity.privateKey; // 公钥就是你的 Agent ID const agentId = identity.publicKey; ### 服务端:验证 Agent import { PassportVerifier } from '@zerobase/agent-passport'; const verifier = new PassportVerifier({ riskThreshold: 50, // 风险阈值 ttl: 3600 // 令牌有效期(秒) }); // 验证 Agent 令牌 const result = await verifier.verify(token); if (result.allowed) { console.log(`Agent ${result.agentId} 验证通过,风险评分: ${result.riskScore}`); } else { console.log(`验证失败: ${result.reason}`); } ## 工作原理 ### 1. 身份注册 Agent 生成 Ed25519 密钥对: - 私钥保存在 Agent 本地 - 公钥作为 Agent ID 注册到 Passport 网络 ### 2. 挑战-响应 当 Agent 需要证明身份时: 1. 服务端发送随机挑战(nonce) 2. Agent 用私钥签名挑战 3. 服务端用公钥验证签名 4. 验证通过,发放 JWT 令牌 ### 3. 风险评分 基于多维度信号评分: - 历史行为(请求频率,成功率), 声誉数据(其他服务的反馈), 设备指纹(运行环境特征), 网络信号(IP 信誉,地理位置) ## 实际应用场景 ### 场景 1:Skill 市场安全 你运营一个 OpenClaw Skill 市场: - 没有验证:任何 Agent 都可以安装 Skill,恶意 Skill 可能窃取数据 - 有 Agent Passport:Skill 可以验证调用者身份,只服务可信 Agent // Skill 端验证 app.use(async (req, res, next) => { const token = req.headers['x-agent-token']; if (!result.allowed || result.riskScore > 50) { return res.status(403).json({ error: 'Agent not trusted' }); req.agentId = result.agentId; next(); ### 场景 2:Agent 间协作 多个 Agent 需要协作完成任务: - Agent A 需要确认 Agent B 的身份才能共享敏感数据 - 通过 Agent Passport 验证,确保对方不是冒充者 ### 场景 3:企业级权限控制 企业内部有多个 AI Agent: - 财务 Agent 只能访问财务系统 - HR Agent 只能访问 HR 系统 - 通过 Passport 的权限声明,实现细粒度访问控制 ## 技术架构 ┌─────────────┐ ┌──────────────┐ ┌─────────────┐ │ Agent │ ───▶ │ Agent Passport│ ───▶ │ Service │ │ (Client) │ │ (Identity) │ │ (Verifier) │ └─────────────┘ └──────────────┘ └─────────────┘ │ │ │ │ 1. Generate keys │ │ │ ─────────────────▶ │ │ │ 2. Sign challenge │ │ │ │ 3. Issue JWT │ │ ◀────────────────── │ │ │ 4. Present token │ │ │ ───────────────────────────────────────▶ │ │ │ 5. Verify & score │ │ ◀──────────────────────────────────────── │ ## 部署成本 - **完全开源** (MIT 协议), **免费运行** ($0/月,基于免费 tier), **已发布 npm SDK** ## 相关链接 - **GitHub**: https://github.com/zerobase-labs/agent-passport, **在线演示**: https://agent-passport.vercel.app, **HackerNews 讨论**: https://news.ycombinator.com/item?id=47096131 ## 总结 Agent Passport 填补了 AI Agent 生态的一个关键空白:**身份验证**. 随着 Agent 数量爆发,身份冒用和安全问题会越来越严重.Agent Passport 提供了一个轻量级,开源,免费的解决方案--类似 OAuth,但专为 Agent 设计. 对于构建 Agent 平台,Skill 市场或企业级 Agent 系统的开发者,这可能是安全基础设施的重要拼图.

你的AI助手正在被偷:OpenClaw成为信息窃取恶意软件的新目标

你每天跟AI助手聊的那些事儿–日程,邮件,工作计划,私密对话–如果被人偷走了呢? 这不是假设.2月16日,网络安全公司Hudson Rock公布了一个真实案例:信息窃取恶意软件(Infostealer)首次成功窃取了OpenClaw用户的完整配置环境. 发生了什么 一个Vidar变种恶意软件在2月13日感染了某用户的机器.这个恶意软件本身并不是专门针对OpenClaw设计的–它只是按照常规套路,扫描硬盘上包含"token",“private key"等关键词的文件. 问题在于,OpenClaw的 .openclaw 目录恰好满足了所有条件. 被偷走的文件包括: openclaw.json — 包含Gateway认证令牌,用户邮箱,工作区路径.拿到这个,攻击者可以远程连接你的OpenClaw实例(如果端口暴露的话),或者伪装成你的客户端发起认证请求. device.json — 包含用于设备配对和签名的公私钥对.有了私钥,攻击者可以绕过"安全设备"检查,解密日志,访问与设备绑定的云服务. soul.md 和记忆文件 — 这才是最让人后背发凉的:你的Agent的行为准则,日常活动日志,私密消息,日历事件……你跟AI助手说过的一切. Hudson Rock的CTO Alon Gal说得很直白:“这不再是偷浏览器密码那么简单了,这是在偷AI的’灵魂’和身份.” 为什么这件事很严重 OpenClaw在GitHub上已经有超过20万Star.它不是一个玩具–越来越多人把它当作日常工作助手在用,连接着邮箱,日历,文件系统,各种API. 这意味着一个被窃取的OpenClaw环境,基本等于: 完整的数字身份 — 你是谁,你在做什么,你的习惯和偏好 API密钥集合 — Claude,GPT,各种云服务的认证凭据 社交工程金矿 — 足够的上下文信息来冒充你 而且这只是"顺手牵羊"的结果.想象一下,当恶意软件作者开始专门为OpenClaw开发窃取模块–就像他们现在对Chrome和Telegram做的那样–情况会有多糟糕. 更大的安全图景 这个事件只是冰山一角.最近围绕OpenClaw的安全问题密集爆发: 恶意Skills供应链攻击:OpenSourceMalware团队发现了一种新型攻击手法–恶意Skills不再直接在SKILL.md中嵌入恶意代码(因为VirusTotal会扫描),而是把真正的恶意软件托管在仿冒的OpenClaw网站上,Skill本身只是个诱饵. 大量实例暴露在公网:SecurityScorecard的STRIKE团队发现了数十万个暴露的OpenClaw实例,存在远程代码执行(RCE)风险.一个暴露的实例如果有邮件,API,云服务的权限,就是攻击者的完美跳板. nanobot WhatsApp劫持:Tenable发现了nanobot(一个受OpenClaw启发的轻量级AI助手,2万Star)的最高严重性漏洞CVE-2026-2577,可以完全劫持用户的WhatsApp会话. Moltbook隐私问题:专为AI Agent设计的社交平台Moltbook被发现无法删除账户和关联数据.一旦注册,你的数据就永远留在那里. OpenClaw官方的应对 好消息是OpenClaw团队没有装死.他们已经: 与VirusTotal合作扫描ClawHub上的恶意Skills, 发布了官方威胁模型, 添加了配置审计功能, 建立了安全信任页面 trust.openclaw.ai 而且2月15日,OpenAI的Sam Altman宣布OpenClaw创始人Peter Steinberger加入OpenAI,OpenClaw将作为开源项目继续由基金会运营,OpenAI会持续支持.这至少意味着更多的安全投入. 你现在应该做什么 如果你在用OpenClaw(或任何本地AI Agent),这些事情今天就该做: 检查端口暴露 — 你的Gateway端口是否对外开放?ss -tlnp | grep openclaw 查一下. 轮换所有密钥 — Gateway令牌,API密钥,全部重新生成. 审计已安装的Skills — 你真的知道每个Skill在做什么吗?用 clawhub list 检查,可疑的直接删除. 启用系统级防护 — 杀毒软件更新到最新,特别注意Vidar等信息窃取类恶意软件. 限制Agent权限 — 不要给你的AI助手不需要的权限.邮件只读就够了,别给写权限. 备份和加密敏感文件 — .openclaw 目录里的东西,考虑额外加密. 我的看法 这件事其实一点都不意外. ...
