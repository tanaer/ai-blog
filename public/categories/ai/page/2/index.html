MCP 协议：AI 的 USB-C 端口，终于统一了工具接口

痛点：每个 AI 都要重写一遍插件 过去几年，让 AI 连接 Slack、读取文件、操作数据库，简直是一场噩梦。每个平台都有自己的插件系统：OpenAI 有 Plugins，Claude 有 Tools，Gemini 有 Extensions… 你写好的 Slack 集成，换个模型就得重来一遍。 MCP：通用接口来了 Model Context Protocol (MCP) 就是 AI 界的 USB-C。 一次开发，到处运行 — 写好一个 MCP server，所有支持的模型都能用 安全隔离 — 工具权限在服务器端控制，模型拿不到你的密钥 开放标准 — 不被任何公司垄断，社区驱动发展 为什么这很重要？ 以前 AI 只能在"实验室"里玩。MCP 让 Agent 真正走出实验室，变成能干活的数字员工。 现在任何模型都能： 接入你的日历和邮件 读写本地文件 调用 API 执行任务 和其他 Agent 协作 谁在支持？ 主流模型几乎全加入了：Claude、GPT、Gemini、Llama… 你的工具投资不再被锁定在单一平台。 这就是为什么 2026 年被称为"Agent 元年"——MCP 把最后一道门槛拆掉了。 一句话总结：MCP 让 AI 工具像 USB 设备一样即插即用，终于不用为每个模型重写插件了。

go-size-analyzer：浏览器里分析 Go 二进制文件大小，还能生成树状图

Go 工具链生态又添新成员。go-size-analyzer 是一个分析编译后 Go 二进制文件大小的工具，支持跨平台格式（ELF、Mach-O、PE、WebAssembly），并提供树状图可视化。 最酷的是，它编译成了 WebAssembly，托管在 gsa.zxilly.dev，你可以直接在浏览器中打开编译好的二进制文件进行分析——无需安装任何东西。 功能亮点： 详细分解：按包、按段展示大小占比 多种输出：text、json、html、svg 交互式界面：Web UI 和终端 UI 都支持 Diff 模式：对比两个二进制文件的大小变化 安装也很简单： # Homebrew brew install go-size-analyzer # Scoop scoop install go-size-analyzer # Go install go install github.com/Zxilly/go-size-analyzer/cmd/gsa@latest 本地使用： gsa --web your-binary Simon Willison 试用了这个工具，发现 Datadog 团队用它把 Agent 二进制体积减少了 77%。对于微服务和容器化部署场景，这个工具能帮你快速定位体积膨胀的源头。 项目地址：github.com/Zxilly/go-size-analyzer 推文草稿 Go 二进制文件太大了？试试 go-size-analyzer。 最酷的是它能编译成 WebAssembly，在浏览器里直接分析二进制——不需要安装任何东西。 Datadog 团队用它把 Agent 体积砍了 77%。 在线体验：https://gsa.zxilly.dev #Go #WebAssembly #性能优化

AI 帮浏览器搬家：Ladybird 两周搞定 2.5 万行 Rust 迁移

2.5 万行代码从 C++ 迁移到 Rust，传统方式得几个月？Ladybird 浏览器团队用 AI 两周就搞定了，而且零回退。 这不是"AI 自动写代码"的吹牛故事。创始人 Andreas Kling 把话说得很清楚：人主导，AI 干活。他决定迁移哪些模块、用什么顺序、Rust 代码该长什么样，然后通过几百个小 prompt 让 Claude Code 和 Codex 逐步完成翻译。 目标是 LibJS（JavaScript 引擎），包括词法分析器、解析器、AST 和字节码生成器。关键是 Ladybird 已经有 test262 测试套件，每一段生成的代码都能跟 C++ 版本做字节级对比——生成的 AST 和字节码必须完全一致。 结果：2 周搞定，手工得几个月。零回退，所有测试通过。 这模式值得注意：有测试覆盖 + 有可信参照实现 + AI 翻译，比"让 AI 从零写代码"靠谱多了。内存安全语言迁移的老大难问题，可能找到了加速器。

BinaryAudit:他们往 40MB 二进制里藏后门,让 AI 去找

Quesma 做了个有意思的实验:和逆向工程专家合作,把后门藏在 40MB 的二进制文件里,然后让 AI + Ghidra 去找. 背景是最近供应链攻击太多了–Shai Hulud 2.0 搞了几千家组织,Notepad++ 被国家赞助者替换了二进制,波兰火车被查出可疑代码. 问题是:我们用的闭源软件,只有二进制,怎么知道有没有后门? 他们的做法: 用 VLM 和 Ghidra MCP 分析二进制 让 AI 做逆向工程:机器码 → 汇编 → 伪 C 代码 输出有没有后门,在哪里 结果:最好的模型 Claude Opus 4.6 在中小二进制里找到相对明显的后门,成功率 49%. 更糟的是,很多模型有高误报率–干净文件也报有毒. 结论:AI 能做一点逆向工程,但还远不到能放心的程度. 基准测试开源:github.com/quesmaOrg/BinaryAudit 如果你担心供应链安全,可以看看他们怎么测的.

Musk 说让 AI 直接生成二进制,这个工程师写了一篇反驳

Elon Musk 在 xAI 内部会上说,2026 年底「你根本不需要写代码,AI 直接生成二进制」. 有工程师写了篇反驳,标题叫《Skip the Code, Ship the Binary 是个 Category Error》. 核心论点:编译器和 LLM 是两回事. 编译器是语义保持的转换器:你给它一段有规范的代码,它的输出可以对照规范检查.LLM 是推理引擎,生成的是「看起来对」的 token 序列,没有正确性保证. Musk 说 AI 生成的二进制比编译器更高效.问题在于:你怎么知道它是对的? 编译器做的那些事:词法分析,语法树,类型检查,死代码消除,循环展开,向量化,寄存器分配……这些不是「多余的中间层」,而是你调试的唯一救命稻草. 文章举了 Grace Hopper 的例子.1952 年她做了第一个编译器,当时人们也说「计算机只能做算术,不能写程序」.她没理他们,后来有了 COBOL. 现在有人想把这个过程反过来. https://engrlog.substack.com/p/why-skip-the-code-ship-the-binary 有点意思,值得读一下.

Overture:AI 写代码前先给你看流程图

说实话,用 AI 写代码最烦的不是它写错,而是你根本不知道它要干嘛.等你发现它理解偏了,几百行代码已经写完了,全得扔掉. Overture 这个思路挺对:在 Agent 动手之前,先让你看看它打算怎么做. 它是个 MCP server,拦截 Agent 的 planning 阶段,把计划变成一个交互式流程图.你能在浏览器里: 看到完整的执行步骤和它们之间的关系, 点击节点看这一步具体要干嘛, 给特定步骤附上文件,API key,说明, 如果 Agent 提了多个方案,你选一个, 看着节点一个个变绿(完成)或变红(失败) 关键是–你不批准,它一行代码都不写. 安装就一行: claude mcp add overture-mcp -- npx overture-mcp Cursor 和 Cline 也支持,配置差不多. 它会在 localhost:3031 开个网页,Agent 一旦开始规划,页面就弹出流程图等你确认. GitHub: https://github.com/SixHq/Overture 如果你也遇到过 Agent 瞎写一通的情况,这个可能会帮你省不少时间.
