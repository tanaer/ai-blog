让 AI Agent 帮你迁移 Postgres 到 ClickHouse:一个 Agent Harness 搞定

你的仪表盘越来越慢,Postgres 快被分析查询压垮了.你知道答案是 Postgres + ClickHouse:事务用 Postgres,分析用 ClickHouse. 但迁移?太痛苦了.AI 来帮忙?大多数 AI 辅助迁移产出的都是垃圾. ClickHouse 官方博客刚分享了一个方法:给 AI Agent 配一个"Harness"(工具链框架),让它能有效完成迁移工作. 为什么普通 AI 迁移会失败? 写个玩具 Demo 没问题.真正难的是生产环境: 重构数据模型以适应 OLAP 性能, 把读时聚合改成写时物化视图, 优化 API 里的查询, 改完前端组件还要验证 这不是"能不能写 ClickHouse SQL"的问题,而是"能不能在复杂的应用栈里有效工作"的问题. 三要素:Code + Cadence + Context 1. 一切皆代码(Code) 把整个数据栈当成一个用代码表达的系统–不是散落的 DDL 脚本和 SQL 字符串,而是共享的,带类型的对象: import { MaterializedView, OlapTable, sql } from "@514labs/moose-lib"; // 定义目标表 schema(带类型和注释) interface TargetSchema { id: string; /** 所有评分的平均值 */ average_rating: number; /** 有效评论总数 */ num_reviews: number; } // 在代码中声明 ClickHouse 表 export const targetTable = new OlapTable<TargetSchema>("target_table", { orderByFields: ["id"], }); // 物化视图:把查询时聚合变成预计算 export const mvToTargetTable = new MaterializedView<TargetSchema>({ selectStatement: sql` SELECT ${sourceTable.columns.id}, avg(${sourceTable.columns.rating}) AS average_rating, count(*) AS num_reviews FROM ${sourceTable} GROUP BY ${sourceTable.columns.id} `, selectTables: [sourceTable], targetTable, materializedViewName: "mv_to_target_table", 这样 AI Agent 就在它熟悉的环境(TypeScript + Git)里工作了. ...
