Spider 2.0-DBT 榜首揭秘：Databao 的秘诀不是更强模型，而是把 AI 当 junior 带

JetBrains 的 Databao Agent 在 Spider 2.0-DBT 基准测试拿了第一。有趣的是，他们的 baseline 最初只能解决 25% 的任务——不是靠换模型翻盘，而是把 AI Agent 当成初级工程师来带。 这个 benchmark 测的是什么？ Spider 2.0-DBT 不是简单的 SQL 生成测试。它给 Agent 一个不完整的 dbt 项目，里面有缺失的模型、错误的代码，Agent 需要： 读整个仓库，理解项目结构 找出哪里坏了、哪里缺了 写 SQL 修复或实现新模型 运行 dbt build，看报错，迭代 直到项目构建成功 68 个任务，有些项目有 2500+ 列的表、几十个模型、数千行 SQL。这不是"写个 SELECT 语句"能搞定的——Agent 得像真正的数据工程师一样工作。 为什么 Databao 能赢？ 他们的核心洞察： 可靠性胜过聪明，prompt 买不来可靠性——你得为它设计。 Databao 团队发现，性能提升最大的改动不是换模型，而是： 减少不确定性：限制 Agent 能访问的工具和数据，避免信息过载 升级上下文：提供清晰的项目文档和依赖关系，而不是让 Agent 瞎猜 强制工作流：像 code review 一样约束 Agent 的行为，不让他乱来 清晰的 policy：把一堆混乱的 prompt 重写成明确的规则 这听起来像什么？像在带一个 junior 工程师。 ...

Overture:AI 写代码前先给你看流程图

说实话,用 AI 写代码最烦的不是它写错,而是你根本不知道它要干嘛.等你发现它理解偏了,几百行代码已经写完了,全得扔掉. Overture 这个思路挺对:在 Agent 动手之前,先让你看看它打算怎么做. 它是个 MCP server,拦截 Agent 的 planning 阶段,把计划变成一个交互式流程图.你能在浏览器里: 看到完整的执行步骤和它们之间的关系, 点击节点看这一步具体要干嘛, 给特定步骤附上文件,API key,说明, 如果 Agent 提了多个方案,你选一个, 看着节点一个个变绿(完成)或变红(失败) 关键是–你不批准,它一行代码都不写. 安装就一行: claude mcp add overture-mcp -- npx overture-mcp Cursor 和 Cline 也支持,配置差不多. 它会在 localhost:3031 开个网页,Agent 一旦开始规划,页面就弹出流程图等你确认. GitHub: https://github.com/SixHq/Overture 如果你也遇到过 Agent 瞎写一通的情况,这个可能会帮你省不少时间.
