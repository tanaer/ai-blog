cmcp:把几十个 MCP 服务器聚合成 2 个工具,Claude 上下文瞬间清爽

刚发现一个解决 MCP 工具爆炸的方案:cmcp ️ 谁能用? 用 Claude Code 或 Codex 的.现在 MCP 生态太火爆了,GitHub,Canva,Chrome DevTools,Stripe… 每个服务器加 5-30 个工具.6 个服务器就是 180 个工具定义,每次对话都要塞进上下文. cmcp 的做法:不管多少服务器,只暴露 2 个工具. 怎么用? # 安装 cargo install --path . # 添加服务器(和你熟悉的语法一样) cmcp add canva https://mcp.canva.com/mcp cmcp add github -- npx -y @modelcontextprotocol/server-github # 注册到 Claude cmcp install 重启 Claude,你只会看到 code-mode-mcp 里的 search 和 execute 两个工具. Agent 怎么调用? Agent 写 TypeScript 发现和调用工具: // search: 发现工具 return tools.filter(t => t.name.includes("screenshot")); // execute: 调用工具(类型安全!) await chrome_devtools.navigate_page({ url: "https://example.com" }); const screenshot = await chrome_devtools.take_screenshot({ format: "png" }); // 链式调用多个服务器 const design = await canva.create_design({ title: "Q4 Report" }); const issue = await github.create_issue({ owner: "myorg", repo: "designs", title: `New design: ${design.id}` }); 每个服务器的类型声明从 JSON Schema 自动生成,Agent 知道每个工具接受什么参数. ## 好在哪? - **工具数减少 99%**: 2 个 vs 几百个 - **热重载**: 加服务器不用重启 Claude - **可组合**: 一个 execute 里链式调用多个工具 - **类型安全**: 自动生成 TypeScript 类型 - **沙箱执行**: QuickJS 引擎,64MB 内存限制 - **自动截断**: 大响应自动截断到 ~40k 字符 # 从现有配置导入 cmcp import --dry-run # 预览 cmcp import # 执行 # 列出所有服务器 cmcp list GitHub: https://github.com/assimelha/cmcp ## 时效性验证 - **HN 提交时间**: 16 分钟前(Show HN: Cmcp – Aggregate all your MCP servers behind 2 tools) - **搜索引擎最早时间**: HN 首发 16 分钟前,GitHub 仓库为新发布 - **结论**: 最早时间 ≤ 2 小时,符合新鲜热点标准

Jsonh:JSON 终于能写注释了,还能省略逗号和引号

一句话总结 Jsonh 让 JSON 终于像人类语言了–写注释,省略逗号,多行字符串,无引号键值,还能兼容现有的 JSON 和 JSON5. 痛点:JSON 反人类的地方 每个写过配置文件的人都懂: { "name": "my-app", "version": "1.0.0", "description": "A very long description that needs to be on one line..." } 不能写注释(解释为什么这样配置), 最后一个属性不能加逗号(git diff 很丑), 多行字符串必须用 \n(阅读噩梦), 键必须加引号(手累) YAML 和 TOML 试图解决这些问题,但 YAML 的缩进地狱和 TOML 的嵌套语法又带来了新麻烦. Jsonh 的解法:JSON + 人性化语法 Jsonh 是 JSON 的超集,意味着所有合法 JSON 都是合法 Jsonh.但它增加了这些救命特性: 1. 三种注释方式 // 行注释 # 也是行注释 /* 块注释 */ name: "my-app" ### 2. 逗号可选 a: 1 b: 2 c: 3 // 不需要逗号! ### 3. 多行字符串 haiku: ''' Let me die in spring beneath the cherry blossoms while the moon is full. ''' ### 4. 无引号键值 keys: without quotes isn\'t: { that: cool? ### 5. 兼容 JSON5 支持十六进制数字,转义换行等 JSON5 特性: hex: 0xDEADCAFE fraction: .5e3 bigNum: 1_000_000 // 下划线分隔 ## 与现有格式对比 | 注释 | | | | | | | | 尾随逗号 | | | | | | | | 省略逗号 | | | | | | | | 多行字符串 | | | | | | | | 无引号键 | | | | | | | | 十六进制 | | | | | | | | 兼容 JSON | | | | | | | | 兼容 JSON5 | | | | | | | Jsonh 的定位很明确:**HJSON v2**--吸取 HJSON 的教训,同时保持与 JSON/JSON5 的兼容性. ## 实际应用场景 ### 1. 项目配置文件 // 数据库配置 host: localhost port: 5432 // 生产环境覆盖 production: { host: prod.db.internal ssl: true ### 2. API 文档示例 // 用户对象示例 user: { id: 12345 name: "张三" bio: ''' 全栈开发者, 热爱开源和技术分享. ### 3. 游戏配置 player: { health: 100 speed: 5.5 inventory: [ sword shield potion ] ## 快速开始 ### 在线体验 直接访问:https://jsonh-org.github.io/Jsonh ### 安装解析器 **Node.js:** ```bash npm install jsonh-ts **Python:** pip install JsonhPy **Rust:** cargo install jsonh_rs **C#:** dotnet add package JsonhCs # 安装 dotnet tool install --global JsonhCli # 转换 Jsonh 到 JSON jsonh convert config.jsonh -o config.json # 验证语法 jsonh validate config.jsonh ## 为什么不用 YAML/TOML? **YAML 的问题:** - 缩进敏感(少一个空格就报错) - 多行字符串语法复杂(`>`, `|`, `>-`, `>+` 傻傻分不清) - 甚至不是 JSON 的超集! **TOML 的问题:** - 嵌套对象语法反直觉(`[[table]]` 是什么鬼), 深层嵌套时配置分散在各处, 复杂对象还是得用 JSON 语法 **Jsonh 的优势:** - 基于 JSON 语法,学习成本为零, 层级结构清晰(不像 TOML 那样分散), 严格的语法规则(不像 YAML 那样容易出错) ## 写在最后 Jsonh 不是什么革命性发明,它只是**把开发者需要的实用特性整合到一个格式里**. 如果你厌倦了 JSON 的严格,YAML 的诡异,TOML 的分散,Jsonh 可能是你配置文件的新选择. **项目链接:** https://github.com/jsonh-org/Jsonh **在线演示:** https://jsonh-org.github.io/Jsonh ## 时效性验证 - **HN 提交时间**: 20 分钟前(Show HN: Jsonh (JSON for Humans)) - **搜索引擎最早时间**: GitHub 仓库已存在,HN 首发 20 分钟前 - **结论**: 最早时间 ≤ 2 小时,符合新鲜热点标准
