任务协调

Cord:让 AI Agent 自己决定如何分解任务,动态构建执行树

一句话总结 Cord 让 AI Agent 自己决定如何分解任务–不是预定义工作流,而是 Agent 在运行时动态构建任务树,自动管理依赖,并行执行和上下文传递. 现有框架的问题 多 Agent 框架正在爆发,但它们都在解决错误的问题: LangGraph, 方式=状态机,预定义节点和边, 问题=图是静态的,Agent 无法中途改变分解方式 CrewAI, 方式=角色扮演(研究员,分析师,作家), 问题=角色由开发者预设,Agent 不能动态调整团队 AutoGen, 方式=群聊模式, 问题=无结构,无依赖追踪,协调从对话中涌现,难以预测 OpenAI Swarm, 方式=轻量级 Agent 切换, 问题=线性执行,无并行,无树结构 Claude Tool-use, 方式=单 Agent 循环, 问题=上下文窗口限制,无法并行 共同问题:所有框架都要求开发者预定义协调结构.你决定工作流图,Agent 角色,切换模式.Agent 在你的边界内执行. 但现在的模型(Claude,GPT-4)已经很擅长规划.它们能自然地将问题分解为子问题,理解依赖关系,知道什么时候任务太大需要拆分. 那为什么还要硬编码分解逻辑? Cord 的解法:让 Agent 构建树 Cord 的核心洞察:给 Agent 一个目标,让它自己决定如何分解. 使用示例 cord run "Should we migrate our API from REST to GraphQL? Evaluate and recommend." 一个 Agent 启动,读取目标,决定需要研究才能回答,然后创建子任务: ● #1 [active] GOAL Should we migrate our API from REST to GraphQL? ● #2 [active] SPAWN Audit current REST API surface ● #3 [active] SPAWN Research GraphQL trade-offs for our stack ○ #4 [pending] ASK How many concurrent users do you serve? blocked-by: #2 ○ #5 [pending] FORK Comparative analysis blocked-by: #3, #4 ○ #6 [pending] SPAWN Write migration recommendation blocked-by: #5 ...
