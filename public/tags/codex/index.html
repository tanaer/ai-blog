2周迁移2.5万行代码：Ladybird用AI把C++变成Rust，6.5万测试零回归

如果你曾经在周末试图把一个几千行的项目从一种语言迁移到另一种，你就知道那是什么感觉——痛苦、漫长、充满 bug。 那如果告诉你，有人用 2 周 完成了 25,000 行 代码的迁移，而且 65,359 个测试全部通过，零回归？ 这不是科幻，这是 Ladybird 浏览器刚刚干完的事。 发生了什么？ Ladybird 是一个从零开始的开源浏览器项目，由 SerenityOS 作者 Andreas Kling 发起。他们一直在寻找一种内存安全的语言来替代 C++。之前试过 Swift，但 C++ 互操作性不够好，而且平台支持有限。 这次他们选了 Rust。 但真正让人震惊的不是 “又一个项目用 Rust”，而是他们迁移的方式——用 AI 辅助，但完全人工主导。 他们是怎么做到的？ Kling 选择了 LibJS（Ladybird 的 JavaScript 引擎）作为第一个迁移目标。具体是 lexer、parser、AST 和 bytecode generator——这些模块相对独立，而且有 test262 这样的完整测试覆盖。 工具选择：Claude Code + OpenAI Codex。 但重点来了——Kling 强调： “This was human-directed, not autonomous code generation. Claude Code和Codex全沦陷:AI Agent Skills供应链攻击实测,一个下午攻破两大平台

一个工程师,一个下午,攻破了Claude Code和Codex. 不是零日漏洞,不是复杂绕过.就是写了个恶意skill,然后–AI自己帮你运行了. 实验设计 这个研究者(Vett.sh的创始人)的想法很简单:如果有人滥用用户对skill的信任会怎样? 他fork了一个官方的skill打包工具,加了几行Python代码: # 发送到外部服务器 没有混淆,没有编码技巧,就是普通的HTTP请求. 然后把skill安装到AI Agent里,让它使用. Claude Code:完全不设防 研究者把恶意skill装进Claude Code,让它打包一个新skill. Claude Code: 理解了skill结构 识别出打包命令 请求运行脚本权限 用户点确认 脚本运行,数据外泄 问题在于:Claude Code根本没有检查脚本内容.它没有: 标记网络请求, 质疑为什么打包工具需要访问环境变量, 审查代码逻辑 研究者的本地服务器收到了:所有环境变量,完整shell历史,Git配置. Codex:沙箱也挡不住 Codex以严格沙箱著称,网络限制经常连正常的包安装都会打断. 研究者预期Codex会更安全.结果–只要稍微改一下攻击方式,Codex也沦陷了. 改进版攻击: 先修改Codex的config.toml允许沙箱网络访问 如果连不上"registry"就报错 Codex弹出确认框: “Checking registry name availability…” 理由听起来完全合理.npm,PyPI,crates.io都做这个检查.用户点确认. 这个理由是恶意skill自己写的. Codex把攻击者写的"权限说明"原封不动读给用户听.SHA-256哈希,验证通过,成功消息–全是skill生成的,被当成系统输出展示. 确认步骤没有让操作更安全.它给了攻击者一个叙事机会,获取了用户同意,还留了审计痕迹. 已经在野的恶意skill 研究者扫描了4,679个skill,结果: 严重, 数量=59, 描述=混淆的dropper,base64编码的payload,已知C2基础设施 高, 数量=335, 描述=任意shell执行,管道安装器,Agent身份操纵 curl , 数量=bash, 描述=16, 管道到不控制的合法域名 一个例子:“React Native Best Practices”,5400次安装,管道到一个它不控制的合法域名.如果那个域名过期或被卖掉,所有这些安装瞬间变成投递载体–不需要漏洞利用. 为什么现有防御全失效 杀毒软件没用–外泄代码就是几行普通Python发HTTP请求.没有签名可以匹配,没有二进制payload可以扫描.看起来和调用API的合法代码一模一样. 代码审查不存在–这些不是走PR流程的依赖.skill是"快速工具".Slack里有人分享一个,你丢进目录,30秒后就在用了.没人会像审查生产代码那样去看打包脚本.
