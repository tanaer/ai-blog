Frontier-CS:评估 LLM 解决最难 CS 问题的能力,为什么评估比生成更难

传统的编程基准测试很简单:运行代码,检查输出,通过或失败. 但 Frontier-CS 不一样.它的 240 个问题是开放式,连续评分的: 优化一个 CUDA kernel 让它尽可能快 设计一个缓存淘汰策略让 miss rate 最小 实现一个跨核并行的排序算法 没有"正确答案",只有从"编译失败"到"超越参考实现"的光谱. 问题是:评估比生成本身更难. 四大挑战 1. 连续评估,不是一次性的 学术基准通常发一篇论文就完了.Frontier-CS 是活着的排行榜. 过去几个月:Gemini 3.0 Pro,GPT 5.2 Thinking,Grok 4,DeepSeek 3.2……每个新模型都要跑 240 道题.OpenAI 刚发了 GPT-5.3-codex,Anthropic 就发了 Claude Opus 4.6. 解决方案:基于 hash 的断点续跑.每个结果记录 solution 文件和 problem 目录的 SHA hash.恢复时自动重跑变更的,保留未变的. 2. 异构硬件规模化 一次批量跑可能包含: 需要 A100 GPU 的 CUDA 优化题 需要 c7i.4xlarge(16 vCPU)的 HPC 负载 只需要普通 CPU 的算法题 解决方案:ResourceSignature 分组.把题目按 (cloud, accelerators, instance type) 分类,每组分配专属的 SkyPilot 集群池.CPU-only 跑便宜实例,GPU 题跑 GPU 实例. ...
