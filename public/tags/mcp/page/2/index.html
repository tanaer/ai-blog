Claude Cowork登陆Windows:Anthropic用10天造了一个要干掉SaaS的桌面AI Agent

上周,Anthropic把Claude Cowork正式带到了Windows.这意味着全球70%的桌面用户现在都能用上这个"AI桌面代理"了. 听起来像又一个AI助手?不是.这东西从根本上改变了AI和你电脑的关系. 不是聊天机器人,是真的在你电脑上干活 传统的AI助手,包括ChatGPT和之前的Claude,本质上都是"你问我答"–你描述问题,它给建议,然后你自己动手. Cowork不一样.你说"把我的研究文件夹按主题整理,然后生成一份摘要文档",它会: 读取你的文件 理解每个文件的内容 创建文件夹结构 移动文件 生成一份格式完美的摘要文档 全程你可以去泡杯咖啡. 技术规格:硬核拆解 Cowork底层跑的是Claude Opus 4.6,带beta版100万token上下文窗口.这意味着它能一次性"看到"你项目里几十个文件的全部内容,而不是像其他AI那样只能处理一小段. 核心能力: 本地文件系统访问:直接操作你电脑上的文件,数据不需要上传到云端, 多步骤任务执行:拆解复杂任务为子任务,并行执行, 专业文档生成:Word,Excel,PowerPoint,PDF,带公式,样式,目录, MCP连接器:通过Model Context Protocol接入外部服务, 插件系统:1月30日上线的11个开源插件,覆盖销售,法务,财务,营销,数据分析,开发 价格:Pro $20/月,Max $100/月起,免费用户用不了. 最疯狂的细节:这玩意是Claude Code用10天写出来的 Anthropic官方说法:Cowork的全部代码是Claude Code自己写的,团队从零到发布只用了大约10天. 让一个AI写另一个AI产品,而且是面向消费者的桌面应用–这在软件工程史上算是一个标志性事件.它证明了AI写生产级代码的能力已经不是实验室玩具了. 2850亿美元的恐慌 Bloomberg报道,Cowork发布后软件股出现了2850亿美元的集体抛售.投资者的逻辑很简单:如果一个AI代理能自主完成文件管理,数据分析,文档生成这些工作,那Notion,Asana,Airtable这类工具的护城河在哪? 这不是杞人忧天.想想看: 你用Notion整理笔记?Cowork直接在你本地文件系统上做, 你用Excel处理数据?Cowork帮你从原始数据到分析报告一步到位, 你用PowerPoint做演示?描述一下要求,Cowork生成一个带样式的.pptx SaaS的核心价值–帮你组织和处理信息–正在被AI Agent侵蚀. 对比微软Copilot和Google Gemini 三者定位其实不同: Microsoft Copilot:嵌入Office 365,帮你在Word里写得更快,在Excel里分析更准.本质是增强现有工具. Google Gemini:深度集成Google Workspace,擅长搜索和Gmail.本质是连接Google生态. Claude Cowork:独立运行,直接操作你的文件系统,不依赖任何生态.本质是替代工作流本身. Copilot让你的Word更好用,Cowork让你不需要打开Word. 插件生态是关键 1月30日上线的插件系统才是Cowork真正的杀手锏.11个官方开源插件涵盖了几乎所有白领工种: 销售助手:CRM数据分析,客户跟进, 法律助手:合同审查,条款分析, 财务助手:报表生成,预算追踪, 开发助手:代码审查,文档生成 而且这些插件是开源的,意味着社区可以在此基础上扩展.按照OpenClaw的Skill生态发展速度来推算,Cowork的插件数量可能在几个月内爆发式增长. 我的判断 Claude Cowork代表了AI从"对话"到"行动"的转变.这是2026年AI领域最重要的产品形态创新之一. 但现阶段它更适合个人知识工作者–自由职业者,研究人员,内容创作者.企业级用户还需要等权限管理,审计日志等功能完善. 如果你是付费Claude用户,现在就可以试试Cowork模式.打开Claude Desktop应用,切换到Cowork标签,给它一个复杂任务,看它怎么拆解和执行. 这种"描述目标→AI自主完成"的工作方式,一旦用过就回不去了. Google WebMCP:AI Agent不用再「看截图猜按钮」了,Chrome正式变身Agent操作系统

先说结论 Google刚发布了一个叫 WebMCP(Web Model Context Protocol) 的东西.一句话:网站可以直接告诉AI Agent自己有哪些功能,Agent不用再截图猜了. 这事儿为什么重要?因为现在所有AI浏览器Agent的工作方式,本质上都是一个字–猜. 现在的Agent有多蠢 你用过任何一个AI浏览器Agent就知道,流程大概是这样的: 截一张网页截图 把截图扔给视觉模型 模型说「我看到一个蓝色按钮,坐标大概在(432, 178)」 点过去 按钮移动了5个像素?挂了. 这不是在「使用网站」,这是在「玩扫雷」. 每一步都要传一张高分辨率截图给LLM,延迟高,成本贵,还经常出错.更别说遇到动态渲染,懒加载,弹窗–基本就是灾难. WebMCP怎么解决的 思路非常简单:别让AI猜了,让网站自己说. WebMCP本质上是一个标准协议,让网站把自己的功能以结构化数据暴露出来.AI Agent拿到的不再是一张图片,而是一份「菜单」–我有什么功能,接受什么参数,怎么调用. Google给了两条路: 路径一:HTML声明式 最简单的方式.网页开发者在HTML标签里加两个属性:toolname 和 tooldescription. 比如一个「订机票」表单,Chrome自动把它解析成一个结构化工具,AI Agent直接知道:这个工具叫「book_flight」,需要出发地,目的地,日期三个参数. 零代码改造成本(好吧,几乎零). 路径二:JavaScript命令式 复杂应用用这个.通过 navigator.modelContext.registerTool() 注册工具,定义名称,描述,JSON Schema输入. 核心API就四个方法: registerTool(): 把功能暴露给AI, provideContext(): 给AI额外上下文(比如用户偏好), clearContext(): 清除共享数据,保护隐私 关键点:AI Agent的操作发生在用户当前session里.不需要重新登录,不需要绕过安全头.这解决了一个巨大的工程问题. 性能提升有多大 从截图猜测到结构化交互,三个维度的提升: 延迟:不用等截图上传+视觉模型处理,直接拿JSON, 准确率:结构化数据交互,错误率接近0%(之前靠猜的时候嘛……你懂的), 成本:发送文本schema比发送高清截图便宜太多了 这不是10%,20%的优化,这是范式级别的切换. 安全怎么搞 很多人第一反应:那网站不是被AI随便操作了? Google的设计是权限优先(permission-first).Chrome作为中间人,AI Agent不能直接执行操作.关键动作(比如「AI要帮你订这趟航班」)会弹出用户确认. 说白了:Agent提议,人类拍板. 这个设计比现在那些偷偷摸摸截图操作的方案安全多了. 我的判断 这是MCP生态的「浏览器版」 熟悉MCP(Model Context Protocol)的朋友应该已经看出来了–WebMCP就是把Anthropic搞的那套MCP理念搬到了浏览器里. 不同的是,Google手里有Chrome.全球65%的浏览器市场份额.这意味着一旦WebMCP成为Chrome标准,所有网站都有动力去适配,因为不适配就意味着你的用户没法用AI Agent操作你的产品. 这跟当年移动适配是一个逻辑:不做响应式设计?Google搜索排名给你降权. Agent框架要大洗牌 现在市面上一堆AI浏览器工具–Browser Use,Playwright AI,各种截图+GPT-4V的方案–它们的核心技术路线(截图→视觉理解→模拟操作)可能在1-2年内变成过渡方案. 当然不会一夜之间消失.WebMCP需要网站主动适配,过渡期里截图方案仍然有用.但方向已经很清楚了. ...
