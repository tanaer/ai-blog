一份 Substack 文章让华尔街恐慌：AI 末日论背后的真相

说真的，华尔街这帮人真是太好笑了。 据 The Guardian 报道，一份 Substack 文章就能让 S&P 跌掉 1%，让 Uber、DoorDash 这些明星股暴跌 4-6%，软件股被打回 2025 年 4 月以来的最低点。SPI Asset Management 管理合伙人 Stephen Innes 说得对：“一份 Substack 文章就能撼动市场”——这话听着像嘲讽，但细想其实是惊恐。 为什么恐慌？因为报告说的东西，他们心里其实信。 Citrini Research 这份"末日报告"的核心逻辑是这样的：AI Agent 会消灭所有"中介型"生意。Claude Code 和 Codex 让企业内部就能干掉原本要外包给 SaaS 公司的活儿，Monday.com、Zapier、Asana 这些公司面临的价格战不是"会不会来"，而是"已经来了"。消费者用个人 Agent 替代旅游中介、房产中介，AI Agent 甚至会主动选择加密货币而不是刷 Visa、Mastercard——因为交易成本更低。 这个逻辑有漏洞吗？坦白说，没找到。 真正让人后背发凉的，是"白领失业"那条线。 报告里有一句话特别扎心：“AI 是通用智能，人类无法转向’AI 管理’岗位”。这是什么意思？就是过去技术革命，农民可以变成工人，工人可以变成白领，但白领没法变成"更高级的白领"——因为 AI 已经在那儿了。 （当然，这个类比有点简化了。农民变工人、工人变白领的过程也不是自动发生的，背后有教育普及、产业政策、社会成本。但方向上，报告的质疑是有道理的：AI 的通用性确实跟以前的技术革命不一样。） 然后呢？失业白领涌入零工经济，送外卖、开 Uber，工资被压得更低。报告管这叫"没有自然刹车的反馈循环"。听起来很极端，但你看看现在 DoorDash 和 Uber 的股价——它们也在跌。为什么？因为送外卖的人多了，平台赚的反而少了。 最绝的是 Citrini Research 提出的"Ghost GDP"这个概念。 报告预测 2027 年 S&P 会暴跌 57%，但同时 AI 公司会蓬勃发展，GDP 数据看起来很好看。问题是，“产出从不流入实体经济”——OpenAI 赚了钱，Google 赚了钱，但普通人呢？“Occupy Silicon Valley"运动围堵 AI 公司这个画面，想想还真有点科幻感。 ...

MCP 协议：AI 的 USB-C 端口，终于统一了工具接口

痛点：每个 AI 都要重写一遍插件 过去几年，让 AI 连接 Slack、读取文件、操作数据库，简直是一场噩梦。每个平台都有自己的插件系统：OpenAI 有 Plugins，Claude 有 Tools，Gemini 有 Extensions… 你写好的 Slack 集成，换个模型就得重来一遍。 MCP：通用接口来了 Model Context Protocol (MCP) 就是 AI 界的 USB-C。 一次开发，到处运行 — 写好一个 MCP server，所有支持的模型都能用 安全隔离 — 工具权限在服务器端控制，模型拿不到你的密钥 开放标准 — 不被任何公司垄断，社区驱动发展 为什么这很重要？ 以前 AI 只能在"实验室"里玩。MCP 让 Agent 真正走出实验室，变成能干活的数字员工。 现在任何模型都能： 接入你的日历和邮件 读写本地文件 调用 API 执行任务 和其他 Agent 协作 谁在支持？ 主流模型几乎全加入了：Claude、GPT、Gemini、Llama… 你的工具投资不再被锁定在单一平台。 这就是为什么 2026 年被称为"Agent 元年"——MCP 把最后一道门槛拆掉了。 一句话总结：MCP 让 AI 工具像 USB 设备一样即插即用，终于不用为每个模型重写插件了。

Ladybird 两周移植 25000 行代码到 Rust，AI 辅助的教科书级案例

Ladybird 浏览器最近干了件狠事——把 JavaScript 引擎从 C++ 移植到 Rust，25000 行代码，两周搞定。按传统方式这得折腾好几个月。 有意思的不是速度，是方法。Andreas Kling 没让 AI 自主写代码，而是用了"人机协作"模式：他决定移植顺序和代码风格，Claude Code 和 Codex 负责翻译。几百个小 prompt，一步步把 lexer、parser、AST、字节码生成器全搬过去。 关键是验证。他们要求 Rust 版和 C++ 版的输出必须字节级一致。靠着 test262 测试套件，最终零回归——每个 AST、每条字节码都跟原版一模一样。 这事儿说明两件事：一是 AI 真的能干重活，前提是你得有好的测试覆盖；二是 Rust 在浏览器领域的吸引力越来越大，内存安全不是闹着玩的。Ladybird 这波算是给"AI 辅助大型重构"打了个样。

Simon Willison 的新项目：用智能体工程对抗 Vibe Coding

Simon Willison 又开新坑了——这次是"Agentic Engineering Patterns"，专门收集编码智能体的最佳实践。 他把现在 AI 编程分成两派：一派是"vibe coding"，完全不看代码让 AI 乱写；另一派是"Agentic Engineering"，专业工程师用 AI 放大自己的能力。很明显，他站后者。 这个项目的定位很有意思：不是博客文章，也不是传统书籍，而是"Guide + Chapter"格式。Guide 是主题集合，Chapter 是可以持续更新的章节——解决了博客内容容易过时的问题。 前两章已经上线： “Writing code is cheap now” 探讨代码成本归零后，我们的工作直觉该怎么调整 “Red/green TDD” 讲测试驱动开发怎么让智能体写得更稳 Willison 还特意强调：内容全是他自己写的，LLM 只负责校对和生成示例代码。这态度挺硬核的，但也说明他对这事儿多认真。 最骚的是，整个 Guide/Chapter 的技术实现，几乎都是 Claude Opus 4.6 在 Claude Code 里写的。用智能体搭建智能体工程指南，这递归玩得溜。 他说打算每周更新 1-2 章，没说什么时候结束——因为这领域本身就还在快速演化。值得关注。

AI 两周搞定浏览器引擎重写，这事儿有点意思

说实话，看到这个标题我第一反应是"又来个 AI 吹牛逼的"。 结果仔细一看，还真不是那么回事。 Ladybird 这个独立浏览器项目，用 AI 辅助把 JavaScript 引擎从 C++ 翻译成了 Rust。不是让 AI 自动写代码，是人工主导翻译——人类决定翻什么、怎么翻，AI 负责干脏活累活。 效果呢？25000 行 Rust 代码，2 周搞定。同样的工作人工做估计得几个月。 关键是没有翻车。test262 的 52898 个测试，Ladybird 自己的 12461 个回归测试，全部通过，零错误。还做了字节级验证，确保 Rust 和 C++ 编译器输出完全一致。 这事儿最有意思的不是"AI 能写代码"，而是展示了一种新的技术债清理方式。传统重写基本等于重做一遍，现在可以让人机协作：人类把控架构和方向，AI 处理机械翻译。 当然，代码还带着浓浓的"C++ 味儿"，不是地道的 Rust。但人家说了，第一要务是兼容性，后续慢慢优化。这思路挺务实的。 独立浏览器本来就难，选择 Rust 本身就说明对安全性的重视。现在再加上 AI 辅助，感觉这个项目可以走得更远了。 说实话，这种"人工 + AI"的协作模式，比那些"AI 自动生成一切"的噱头实在多了。

BinaryAudit:他们往 40MB 二进制里藏后门,让 AI 去找

Quesma 做了个有意思的实验:和逆向工程专家合作,把后门藏在 40MB 的二进制文件里,然后让 AI + Ghidra 去找. 背景是最近供应链攻击太多了–Shai Hulud 2.0 搞了几千家组织,Notepad++ 被国家赞助者替换了二进制,波兰火车被查出可疑代码. 问题是:我们用的闭源软件,只有二进制,怎么知道有没有后门? 他们的做法: 用 VLM 和 Ghidra MCP 分析二进制 让 AI 做逆向工程:机器码 → 汇编 → 伪 C 代码 输出有没有后门,在哪里 结果:最好的模型 Claude Opus 4.6 在中小二进制里找到相对明显的后门,成功率 49%. 更糟的是,很多模型有高误报率–干净文件也报有毒. 结论:AI 能做一点逆向工程,但还远不到能放心的程度. 基准测试开源:github.com/quesmaOrg/BinaryAudit 如果你担心供应链安全,可以看看他们怎么测的.

Musk 说让 AI 直接生成二进制,这个工程师写了一篇反驳

Elon Musk 在 xAI 内部会上说,2026 年底「你根本不需要写代码,AI 直接生成二进制」. 有工程师写了篇反驳,标题叫《Skip the Code, Ship the Binary 是个 Category Error》. 核心论点:编译器和 LLM 是两回事. 编译器是语义保持的转换器:你给它一段有规范的代码,它的输出可以对照规范检查.LLM 是推理引擎,生成的是「看起来对」的 token 序列,没有正确性保证. Musk 说 AI 生成的二进制比编译器更高效.问题在于:你怎么知道它是对的? 编译器做的那些事:词法分析,语法树,类型检查,死代码消除,循环展开,向量化,寄存器分配……这些不是「多余的中间层」,而是你调试的唯一救命稻草. 文章举了 Grace Hopper 的例子.1952 年她做了第一个编译器,当时人们也说「计算机只能做算术,不能写程序」.她没理他们,后来有了 COBOL. 现在有人想把这个过程反过来. https://engrlog.substack.com/p/why-skip-the-code-ship-the-binary 有点意思,值得读一下.
