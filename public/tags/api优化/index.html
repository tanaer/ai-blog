你的 AI 账单可能多付了一倍

你的 AI 账单可能多付了一倍 昨晚睡得晚，躺在床上刷手机，看到一条推送差点没睡着。OpenAI 有个功能叫 Prompt Caching，能帮你省 90% 的钱。我寻思我这天天调 API，是不是亏大了？爬起来算了算账，还真亏了不少。这事儿得说说。其实原理挺简单，就是你问模型的问题它会记下来，下次再问一样的直接给你答案，不用重新算。打个比方，你去饭店点菜，每次都说"不要葱姜蒜"，服务员记住了，下次你去不用再说，直接就给你做了。模型也一样，你每次传的系统提示词，“你是一个助手，用中文回答…“这几千个字，每次都要重新处理一遍，但其实都是一样的，多浪费啊。Prompt Caching 就是让模型记住，下次遇到同样的直接跳过。 说到这儿我想起前两年，我刚接触 AI 的时候根本不知道这个，写了个客服机器人，每天调用好几万次，系统提示词三千字每次都原封不动传过去。现在想想那钱烧得，心都在滴血。 我给你算笔账你就明白了。假设你的系统提示词 2000 tokens，调用 10 万次。没有缓存的话 2000 × 100,000 = 2 亿 tokens，按 GPT-4 的价格大概 2000 美元。有缓存呢，第一次正常算，后面 99999 次缓存命中，价格直接打一折甚至有些场景免费，最后可能就 200 美元。差了十倍，一年下来省个两万美金，这还是小项目。我有个朋友做电商客服的，每天 10 万次调用，系统提示词加产品手册 15000 tokens，以前一个月 4 万多，用了缓存之后 6000，一年省了 46 万。他跟我说这钱够雇两个员工了，我说你雇啥员工，这钱给我，我帮你写提示词。开玩笑的，但确实这省下来的钱干啥不香？ 好消息：根本不用管。OpenAI 自动帮你做，只要两个条件：提示词超过 1024 tokens，内容有重复。就这样，你啥都不用改，系统自己搞定。坏消息是你得会"排兵布阵”，不是真的打仗，是安排提示词顺序。 记住一句话：静态的放前面，动态的放后面。啥意思呢？看个例子。错误的做法是把用户问题放在最前面，用户问题每次不一样，缓存永远不会命中，因为前缀变了 hash 值不对了。正确的做法是把系统提示词放前面，每次都一样，缓存命中率 99%。就这么简单，改个顺序省钱效果立竿见影。我第一次知道这个的时候愣了半天，这么简单？就这？还真就这，但很多人不知道，包括以前的我。 三种场景最划算：对话机器人，系统提示词固定用户问题不同，完美契合；RAG 应用，检索的文档很长每次都传一遍，不缓存就是烧钱；Agent 系统，工具定义一大堆每次调用都传，缓存能省一大笔。如果你的应用超过 50% 的内容会重复，就一定要用，不用就是亏。 有坑，我踩过。缓存会过期，大概 5-10 分钟，如果两次调用间隔太久缓存就没了，得重新算。必须是精确匹配，差一个字都不行，包括空格包括换行，所以提示词要固定，别今天改一下明天改一下，缓存永远命中不了。我之前就犯过这个错，以为开了缓存结果每天都在调提示词，根本没省到钱，后来才发现稳定下来再上线，别瞎折腾。 看 API 返回值就知道生效没，有个字段叫 cached_tokens，是 0 就没命中，有数字就说明生效了。我一般会打个日志每天看一眼心里有数，如果命中率低于 50% 说明提示词结构有问题得调整。 说实话，以前做项目算算 API 成本就吓退了，觉得不划算。现在成本降 90%，很多项目突然就可行了。前两天有个想法，做个自动写周报的工具，以前算了一下一个月 API 费用要好几百觉得没必要，现在几十块搞定，立马就做了。创意不再被成本扼杀，这是最大的变化。 ...
