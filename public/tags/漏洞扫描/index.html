Filippo：关掉 Dependabot，它是个噪音机器

一句话总结 Go 安全专家 Filippo Valsorda 建议关掉 Dependabot——它制造了大量无意义的警报和 PR，让你感觉在做安全工作，实际上只是在处理噪音。用 govulncheck + 定期 CI 测试才是正道。 发生了什么？ Filippo 最近发布了一个安全修复（filippo.io/edwards25519 库的 MultiScalarMult 方法），结果 Dependabot 的表现让他彻底失望： 问题 1：数千个无意义的 PR Dependabot 向未受影响的仓库提交了数千个 PR，要求更新 edwards25519 库。这些项目根本不使用受影响的 MultiScalarMult 方法。 问题 2：荒谬的 CVSS 评分 警报附带了一个"胡编乱造"的 CVSS v4 分数，以及一个 73% 的兼容性评分——基于更新在生态系统中造成的破坏程度。 问题 3：完全错误的警报 Wycheproof 仓库甚至收到了警报，但它根本不导入受影响的包，只导入了未受影响的子包。 $ go mod why -m filippo.io/edwards25519 # filippo.io/edwards25519 github.com/c2sp/wycheproof/tools/twistcheck filippo.io/edwards25519/field # 未受影响的子包！ 结果：Wycheproof 团队直接关掉了 Dependabot。 为什么 Dependabot 会这样？ Dependabot 的核心问题是：它只看模块版本，不看实际代码路径。 你的项目依赖了 edwards25519 v1.1.0 有漏洞，需要升级到 v1.1.1 Dependabot 就给你发警报、开 PR 但它不管你是否真的调用了那个有漏洞的方法。 ...
