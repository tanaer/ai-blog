AI 写了一个 FreeBSD 内核驱动，还能连 Wi-Fi

2026年2月25日 · 3 分钟 · 427 字 · AI 极客前沿
⚡ 快捷操作 🚀 发到 X

旧电脑别扔，让 AI 给它写个驱动。

一个尘封的 MacBook Pro#

Vladimir Varankin 有一台 2016 年的 MacBook Pro，屏幕有 flexgate 问题，一直躺在柜子里吃灰。FreeBSD 15 发布后，他决定把这台电脑改造成 FreeBSD 实验机——直到发现 Wi-Fi 用不了。

2016 MacBook Pro 用的 Broadcom BCM4350 芯片，FreeBSD 原生不支持。论坛上的标准方案是跑一个叫 wifibox 的小 Linux 虚拟机，把 PCI 设备直通给 Linux。

但 Varankin 想的是：能不能写一个原生的 FreeBSD 驱动？

第一次尝试：直接移植，直接翻车#

2026 年了，“把代码从 A 移植到 B” 的第一反应当然是让 AI 干。

他克隆了 Linux 的 brcmfmac 驱动源码，让 Claude Code 照着 FreeBSD 已有的 iwlwifi 驱动（Intel 网卡）移植。Claude 信心满满地说可以。

模块确实编译通过了——但没有任何反应。因为他测试的虚拟机里根本没有那张网卡。

等他把 PCI 设备加进去，真机测试：kernel panic。

Claude 开始疯狂打补丁，到处加 #ifdef __FreeBSD__，抱怨 LinuxKPI 缺这少那。补丁越打越多，驱动离能用的距离却一点没缩短。

战略转型：先写规范，再写代码#

Varankin 意识到方向错了。brcmfmac 驱动代码量很大，支持好几代芯片、多种接口。但他的需求很窄：就这一颗芯片，只要 PCI，只要能当 Wi-Fi 客户端。

他换了个思路：

  1. 让 AI 先写一份详尽的技术规范，解释 brcmfmac 到底是怎么工作的
  2. 要求写得足够详细，“到比特级别”，假设读者要在"净室环境"重新实现

然后他开了多个 AI 会话交叉验证：

  • 让 Codex 对比规范和源码，找出不一致
  • 让 Opus 验证 Codex 的修改建议
  • 顺便测试了多个模型：Opus 4.5、Opus 4.6、Codex 5.2、Gemini 3 Pro

结论：Gemini 幻觉最多。

从规范到代码#

有了规范，Varankin 开了一个全新项目，只有这份规范文档。他让 Pi Agent：

  1. 先提问，列出所有必须明确的技术决策
  2. 把这些决策记录在项目的 AGENTS.md 里
  3. 写代码时必须引用这些决策文档

关键决策包括：

  • 驱动要不要放进内核源码树？
  • 用 C 还是其他语言？
  • 依赖 LinuxKPI 吗？
  • 里程碑怎么划分？

结果：一个能工作的原生 FreeBSD Wi-Fi 驱动诞生了。

GitHub 项目：narqo/freebsd-brcmfmac

目前支持 WPA2-PSK，2.4GHz 和 5GHz 都能用。代码主要由 AI 编写，人工监督很少——Varankin 自己说是"vibe coding over many coding sessions"。

这意味着什么#

  1. AI 能写内核代码，但不是"直接扔给它就行"
  2. 中间产物很重要：规范文档、决策记录、交叉验证
  3. 多 Agent 协作：不同的模型做不同的事，互相校验
  4. 人类仍然是架构师：知道什么时候该停下来换方向

当然，社区里也有质疑声音：“让 AI 写内核模块是疯狂的想法，一个糟糕的模块可能让系统崩溃，还可能带来各种安全漏洞。”

Varankin 的回应很简单：这是我的实验项目，跑在我自己的机器上。

这个故事告诉我们：AI 编程的上限，取决于你如何分解问题。直接让它"写一个驱动"和"先写规范再按规范实现"，是完全不同的两件事。
