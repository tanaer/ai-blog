AGENTS.md到底有没有用?一篇论文引爆开发者社区大战

如果你最近在用 Claude Code,Codex 或者 OpenClaw 之类的 AI 编程工具,你大概率见过一个叫 AGENTS.md(或 CLAUDE.md)的文件.它的作用很简单:告诉 AI Agent 这个项目的规矩–用什么框架,怎么跑测试,有什么坑要避. 听起来很合理对吧? 然后一篇 arXiv 论文说:这玩意不但没用,还让 Agent 变蠢了. 论文说了啥 这篇来自学术界的论文 “Evaluating AGENTS.md: Are Repository-Level Context Files Helpful for Coding Agents?” 做了一个挺严谨的实验: 在 SWE-bench(业内标准的代码修复基准测试)上跑了多个 Agent + 多个 LLM 的组合 分别测试了三种情况:没有上下文文件,LLM 自动生成的上下文文件,开发者手写的上下文文件, 结果:, LLM 生成的文件:任务成功率平均下降 3% 开发者手写的文件:平均提升 4%,但在不同模型上表现不一致(Sonnet 4.5 反而下降 2%) 所有情况下推理成本增加超过 20% 论文的结论是:上下文文件里的"不必要要求"让任务变难了,建议只写最小化的必要信息. HN 社区炸了 这篇论文在 Hacker News 上拿到 146 分,94 条评论,评论区基本是一边倒的"你们测错了". 最核心的反驳来自一位高赞评论者: “4% 的提升是巨大的!一个简单的 markdown 文件就能带来 4% 的改善,这说明它是必须有的.” 他指出了论文的一个关键盲区:SWE-bench 测试的都是公开的 GitHub 项目,而这些项目里的 AGENTS.md 大多来自"最近的小型 vibecoding 项目",质量参差不齐.真正需要 AGENTS.md 的场景是大型私有代码库–那些有复杂部署流水线,团队约定,历史债务的项目. ...

一个配置错误搞瘫半个互联网:Cloudflare BGP事故暴露了我们不敢面对的真相

一个周日下午,半个互联网消失了 2026年2月16日,美东时间下午1点15分. 一个Cloudflare工程师在弗吉尼亚州阿什本数据中心执行了一次"常规配置更新".几分钟后,全球互联网开始抽搐. X(原Twitter)时间线一片空白,Grok同时挂掉.AWS的US-East-1区域出现"间歇性连接降级".电商网站结账流程中断.数千个依赖Cloudflare CDN的中小型网站直接返回错误页面. Downdetector在4小时内录得超过11,000条X平台故障报告,峰值出现在下午4:20.美国,英国,欧洲部分地区用户首当其冲. 4个小时后,工程师们才完全恢复服务. 这不是一次黑客攻击.不是DDoS.甚至不是什么复杂的技术故障. 这是一个BGP路由配置错误.一个. BGP:互联网的GPS,也是最脆弱的那根弦 BGP(Border Gateway Protocol,边界网关协议)是互联网的"路由导航系统".它告诉数据包:“从A到B,走这条路.“全球互联网的流量路由几乎完全依赖这个上世纪80年代设计的协议. 问题在于:BGP本质上是基于信任的.一个网络宣布"我能到达这些IP地址”,其他网络就信了.没有严格的验证机制(虽然RPKI等改进在推进,但普及率还很低). 当Cloudflare阿什本节点的BGP路由表被错误配置后,坏的路由信息像病毒一样传播到上游Provider.正常应该流经Cloudflare全球300多个数据中心的流量,被错误导向,丢弃,或者在网络中打转. 最讽刺的是:Cloudflare自己的状态页也挂了. 工程团队40分钟内定位了根因,但回滚比部署困难得多–因为损坏的路由表已经被上游Provider缓存了.你没法简单地"撤回"一条BGP公告,它已经在整个互联网中扩散. 多米诺骨牌:AWS和X为什么也倒了 这次事故最值得警惕的不是起因,而是传播方式. AWS承载着全球约三分之一的云基础设施.虽然AWS有独立的网络骨干,但大量客户流量在到达AWS之前,会先经过Cloudflare做DDoS防护和CDN加速.当Cloudflare路由失效后,海量的重试连接和误导数据包涌入AWS边缘路由器,直接把几台路由器打爆了. X平台更惨.自从马斯克收购后大幅削减基础设施成本,X同时依赖Cloudflare做边缘安全,AWS做后端计算.两个Provider同时出问题,X基本没有fallback方案.结果就是:将近3小时无法发帖,刷时间线,收发私信. Bluesky和Threads的用户量在这几个小时内暴涨.有评论员说:“全球最有钱的科技人士拥有的平台,被一行配置搞挂了.” 表面分布式,实际深度中心化 普林斯顿大学计算机科学教授Jennifer Rexford(前FCC技术顾问委员会成员)的话一针见血: “我们构建了一个看起来分布式,实际上深度中心化的系统.每次这类宕机发生,我们都进行同样的’韧性’讨论,然后什么都没变–因为经济激励全都指向集中化.” 几个数字让你感受一下: Cloudflare处理全球约**20%的网络流量, AWS占据云基础设施市场约31%**的份额, 加上Azure和GCP,三家公司控制了全球超过60%的云计算 当这些巨头之间还存在深度依赖关系时(比如用Cloudflare做CDN + AWS做计算),一个节点的故障就能引发系统性崩溃. 这不是分布式系统.这是一个伪装成分布式的中心化系统. 这事以前发生过,以后还会发生 回顾一下历史: 2024年7月:CrowdStrike更新导致全球Windows蓝屏,航空,银行,医疗系统瘫痪, 2021年10月:Facebook(Meta)BGP配置错误,Facebook/Instagram/WhatsApp全球宕机6小时, 2021年6月:Fastly CDN故障,Amazon,Reddit,CNN等大面积下线, 2020年11月:AWS US-East-1宕机,半个互联网跟着挂 每次事故后都有一轮关于"互联网韧性"的讨论.然后呢?什么都没变. 原因很简单:集中化是经济理性的选择.用Cloudflare比自建CDN便宜100倍,用AWS比自建数据中心快100倍.没有哪个CTO会为了理论上的"韧性"去增加10倍的基础设施成本. 真正的问题:我们能做什么? 作为技术从业者,我认为有几个层面值得思考: 1. 架构层面:多云不是口号 如果你的服务100%跑在AWS上,Cloudflare做CDN,那你的SPOF(单点故障)比你以为的多得多.真正的多云意味着: 关键服务在至少两个云Provider上有热备, CDN/安全层有备用方案(比如Cloudflare + Fastly双活), DNS做好多Provider解析 成本会增加30-50%,但对于核心业务来说这是保险费. 2. 协议层面:推进RPKI和BGP安全 RPKI(Resource Public Key Infrastructure)可以验证BGP路由公告的合法性,但全球采用率还不到50%.这需要整个行业的推动,不能靠单个公司. 3. 监管层面:互联网基础设施需要更高标准 Cloudflare,AWS这些公司本质上是互联网的"公用事业”,但它们承担的监管义务远低于电力,电信等传统公用事业.也许是时候讨论基础设施层面的韧性标准了. 4. 认知层面:接受"零宕机"是幻觉 任何足够复杂的系统都会出故障.问题不是"会不会挂",而是"挂了之后多快能恢复".设计系统时,优雅降级比防止故障更重要. 我的看法 这次Cloudflare事故不是黑天鹅–它是灰犀牛.每个基础设施工程师都知道BGP的脆弱性,每个云架构师都知道Provider集中度的风险.但经济激励和技术惯性让我们选择性忽视. ...

Qwen3.5横空出世:397B参数只激活17B,推理速度暴涨19倍,开源AI格局要变了

2月16日,阿里通义千问团队放了个大招–Qwen3.5-397B-A17B,Qwen3.5系列的第一个开源模型. 这名字里藏着关键信息:总参数397B,但每次推理只激活17B.换句话说,你用的是一个"大脑容量"接近4000亿参数的模型,但每次思考只调动170亿参数的计算量. 这意味着什么?推理速度暴涨.阿里给出的数据是:在256K token长上下文场景下,吞吐量是Qwen3-Max的19倍;在标准32K场景下也有8.6倍的提升. 更重要的是,它以Apache 2.0协议完全开源. 架构上到底改了什么? Qwen3.5的核心创新是引入了Gated Delta Network–一种混合线性注意力机制,配合细粒度的稀疏MoE(Mixture of Experts)架构. 传统Transformer的注意力机制计算复杂度随序列长度平方增长,这是长上下文处理的老大难问题.Delta Network通过线性注意力变体大幅降低了这个开销,而"Gated"机制则让模型能动态决定哪些信息值得关注. MoE的粒度也值得注意.397B总参数只激活17B,意味着专家数量极多,每个专家极小.这种设计的好处是:不同类型的任务可以由高度专业化的"小专家"处理,既保证了质量,又控制了计算成本. 跑分怎么样?说人话 先说强项: 指令遵循:IFBench 76.5,MultiChallenge 67.6,都是业界最高, Agent任务:TAU2得分86.7,仅次于Claude 4.5 Opus(91.6)和GPT-5.2(87.1), 多语言:支持201种语言和方言,词表从15万扩到25万, 数学视觉:MathVision 88.6,ZEROBench 12分超过所有前沿模型, 文档理解和OCR:多项测试领先 再说短板–这一点阿里自己也没藏着: 纯推理:AIME26得分91.3,GPT-5.2是96.7,Claude 4.5 Opus是93.3 代码:LiveCodeBench 83.6,GPT-5.2是87.7 通用图像理解:MMMU 85,Gemini 3 Pro是87.2 简单总结:在效率和多模态方面达到了第一梯队,但在纯推理和硬核编程上还差一口气.不过考虑到它是开源的,推理成本极低,这个性价比已经非常炸裂了. Google WebMCP:AI Agent不用再「看截图猜按钮」了,Chrome正式变身Agent操作系统

先说结论 Google刚发布了一个叫 WebMCP(Web Model Context Protocol) 的东西.一句话:网站可以直接告诉AI Agent自己有哪些功能,Agent不用再截图猜了. 这事儿为什么重要?因为现在所有AI浏览器Agent的工作方式,本质上都是一个字–猜. 现在的Agent有多蠢 你用过任何一个AI浏览器Agent就知道,流程大概是这样的: 截一张网页截图 把截图扔给视觉模型 模型说「我看到一个蓝色按钮,坐标大概在(432, 178)」 点过去 按钮移动了5个像素?挂了. 这不是在「使用网站」,这是在「玩扫雷」. 每一步都要传一张高分辨率截图给LLM,延迟高,成本贵,还经常出错.更别说遇到动态渲染,懒加载,弹窗–基本就是灾难. WebMCP怎么解决的 思路非常简单:别让AI猜了,让网站自己说. WebMCP本质上是一个标准协议,让网站把自己的功能以结构化数据暴露出来.AI Agent拿到的不再是一张图片,而是一份「菜单」–我有什么功能,接受什么参数,怎么调用. Google给了两条路: 路径一:HTML声明式 最简单的方式.网页开发者在HTML标签里加两个属性:toolname 和 tooldescription. 比如一个「订机票」表单,Chrome自动把它解析成一个结构化工具,AI Agent直接知道:这个工具叫「book_flight」,需要出发地,目的地,日期三个参数. 零代码改造成本(好吧,几乎零). 路径二:JavaScript命令式 复杂应用用这个.通过 navigator.modelContext.registerTool() 注册工具,定义名称,描述,JSON Schema输入. 核心API就四个方法: registerTool(): 把功能暴露给AI, provideContext(): 给AI额外上下文(比如用户偏好), clearContext(): 清除共享数据,保护隐私 关键点:AI Agent的操作发生在用户当前session里.不需要重新登录,不需要绕过安全头.这解决了一个巨大的工程问题. 性能提升有多大 从截图猜测到结构化交互,三个维度的提升: 延迟:不用等截图上传+视觉模型处理,直接拿JSON, 准确率:结构化数据交互,错误率接近0%(之前靠猜的时候嘛……你懂的), 成本:发送文本schema比发送高清截图便宜太多了 这不是10%,20%的优化,这是范式级别的切换. 安全怎么搞 很多人第一反应:那网站不是被AI随便操作了? Google的设计是权限优先(permission-first).Chrome作为中间人,AI Agent不能直接执行操作.关键动作(比如「AI要帮你订这趟航班」)会弹出用户确认. 说白了:Agent提议,人类拍板. 这个设计比现在那些偷偷摸摸截图操作的方案安全多了. 我的判断 这是MCP生态的「浏览器版」 熟悉MCP(Model Context Protocol)的朋友应该已经看出来了–WebMCP就是把Anthropic搞的那套MCP理念搬到了浏览器里. 不同的是,Google手里有Chrome.全球65%的浏览器市场份额.这意味着一旦WebMCP成为Chrome标准,所有网站都有动力去适配,因为不适配就意味着你的用户没法用AI Agent操作你的产品. 这跟当年移动适配是一个逻辑:不做响应式设计?Google搜索排名给你降权.
