AGENTS.md到底有没有用?一篇论文引爆开发者社区大战

一篇arXiv论文称AGENTS.md文件反而降低编码Agent成功率,增加20%推理成本.但HN开发者社区集体反驳:你们测的东西根本不对.这场争论揭示了AI编程工具的核心矛盾.
2026年2月17日 · 1 分钟 · 197 字 · AI 极客前沿
⚡ 快捷操作 🚀 发到 X

如果你最近在用 Claude Code,Codex 或者 OpenClaw 之类的 AI 编程工具,你大概率见过一个叫 AGENTS.md(或 CLAUDE.md)的文件.它的作用很简单:告诉 AI Agent 这个项目的规矩–用什么框架,怎么跑测试,有什么坑要避.

听起来很合理对吧?

然后一篇 arXiv 论文说:这玩意不但没用,还让 Agent 变蠢了.

论文说了啥#

这篇来自学术界的论文 “Evaluating AGENTS.md: Are Repository-Level Context Files Helpful for Coding Agents?” 做了一个挺严谨的实验:

  • 在 SWE-bench(业内标准的代码修复基准测试)上跑了多个 Agent + 多个 LLM 的组合
  • 分别测试了三种情况:没有上下文文件,LLM 自动生成的上下文文件,开发者手写的上下文文件, 结果:, LLM 生成的文件:任务成功率平均下降 3%
  • 开发者手写的文件:平均提升 4%,但在不同模型上表现不一致(Sonnet 4.5 反而下降 2%)
  • 所有情况下推理成本增加超过 20%

论文的结论是:上下文文件里的"不必要要求"让任务变难了,建议只写最小化的必要信息.

HN 社区炸了#

这篇论文在 Hacker News 上拿到 146 分,94 条评论,评论区基本是一边倒的"你们测错了".

最核心的反驳来自一位高赞评论者:

“4% 的提升是巨大的!一个简单的 markdown 文件就能带来 4% 的改善,这说明它是必须有的.”

他指出了论文的一个关键盲区:SWE-bench 测试的都是公开的 GitHub 项目,而这些项目里的 AGENTS.md 大多来自"最近的小型 vibecoding 项目",质量参差不齐.真正需要 AGENTS.md 的场景是大型私有代码库–那些有复杂部署流水线,团队约定,历史债务的项目.

另一位在生产环境中管理多 Agent 系统的开发者更直接:

“这篇研究衡量的东西就不对.任务完成(PR 是否通过测试)只是一个狭隘的代理指标.我要的不是 Agent 能修 bug,而是它修 bug 的方式要符合团队规范.没有 AGENTS.md,它写出来的代码像一个昨天才入职的人–测试过了,但违反了所有约定.”

还有一位论文作者也出现在评论区做了回应:

“关于人写的 AGENTS.md 的 4% 提升–这个数字如果是一致的改善当然很大,但问题是它在不同模型上表现不稳定.”

这倒是个诚实的回应.

真正的分歧在哪#

吵了半天,真正的分歧其实不是"AGENTS.md 有没有用",而是怎么衡量"有用".

论文用的是 SWE-bench 的任务完成率–能不能通过测试.这是一个可量化,可复现的指标,学术上没问题.

但实际开发中,“通过测试"只是最低标准.一个好的代码变更还要:

  • 符合项目架构风格, 使用正确的 ORM 模式, 遵循团队的测试规范, 不引入后续需要重构的技术债

这些东西测试套件根本管不到,但恰恰是 AGENTS.md 最有价值的地方.

一位评论者总结得很好:

“论文发现上下文文件’鼓励更广泛的探索’–更多的测试,更多的文件遍历.这恰恰是重点.我就是要 Agent 在写测试之前先读测试规范,在建表之前先查 migration 模式.多花 20% 的 token?撤销一个用错 ORM 模式的合并 PR 花的可比这多.”

我的看法#

这篇论文做了一件好事:它让我们认真思考 AGENTS.md 应该怎么写.

实际经验告诉我们:

  1. LLM 自动生成的 AGENTS.md 基本没用–这个论文的发现是对的.让 AI 总结项目然后告诉自己"这是规矩”,等于左脚踩右脚上天.

  2. 好的 AGENTS.md 应该来自失败经验.一位 HN 评论者分享了自己的方法论:只有当 Agent 在某个任务上犯了错,他才会往 AGENTS.md 里加信息,然后回滚重跑验证效果.这才是正确的迭代方式.

  3. 越大的项目越需要.小项目里 Agent 自己就能摸清结构.但当代码库有几十万行,十几个微服务,三种不同的测试框架时,那份"部落知识"(tribal knowledge)就是省下来的几十轮无效探索.

  4. 少即是多.论文最有价值的发现是:冗余信息确实会拖后腿.别把 AGENTS.md 写成第二个 README.只写 Agent 自己猜不到的东西.

给开发者的实操建议#

如果你在用 AI Agent 写代码,我的建议是:


# AGENTS.md 最佳实践

## 该写的
- 项目特有的构建/测试命令(不是标准的那种)
- 非显而易见的架构决策(比如"为什么用 X 而不是 Y")
- Agent 反复犯的错误(每次犯一次加一条)
- 关键的目录结构说明

## 不该写的
- README 里已有的信息, 通用编程规范(Agent 已经知道), 过于详细的代码风格规定(用 linter 解决), 长篇大论的项目历史

**最后一句话:**AGENTS.md 不是给 Agent 的入职手册,是给它的"防坑指南".写你踩过的坑,不写它已经知道的东西.
