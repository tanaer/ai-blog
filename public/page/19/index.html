1亿人的饭碗,春晚一个字没提

今年春晚看了吗? 机器人空翻,醉拳,跑酷,十几台机器同时编队起舞.仿生奶奶和孙子演小品,AI视觉特效满屏飞.整台晚会把过去一年中国具身智能最炸裂的Demo全塞进去了,四个小时看下来,弹幕清一色"太牛了"“中国速度"“未来已来”. 我承认动作确实一年比一年丝滑.去年机器人还在笨拙地转手绢,今年已经连续空翻加跑酷了,进步速度快到你觉得在看加速版科幻片. 但看完那个晚上,我翻来覆去睡不着,一直在想一件事– 春晚把中国最先进的AI和机器人,打包成了一场四小时的宣传片. 展示的全是"多炫"“多像人"“迭代多快”,配的旁白永远是:展示中国智造硬实力,科技与传统文化完美融合,人机共舞未来可期. 听着真提气. 但总觉得哪不对. …… 哪不对呢? 它把最有破坏力的东西,包装成了最无害的样子. 你看到机器人打醉拳,觉得好可爱好厉害.你没看到的是,等这项技术再成熟两三年,建筑工地的搬砖工,物流仓库的分拣员,工厂流水线上的操作工,他们的位置会被一个不要社保,不请假,不闹情绪,24小时连轴转的铁家伙顶掉. 你看到AI奶奶和孙子在台上逗乐,觉得好温馨好感人.你没想到的是,客服,初级文案,基础设计,数据标注这些白领岗位,正在以比你想象快得多的速度蒸发. 不是"AI创造新岗位补上缺口"那套说辞能糊弄过去的. 活在减少,钱在变少. 我来算一笔账. 中国制造业从业人员大约1亿,服务业从业人员3.6亿.其中重复性体力劳动加上重复性脑力劳动,保守估计占60%. 如果AI和机器人在未来5-10年替代其中20%的岗位–注意,我只说20%,不是60%–那就是接近1亿人需要重新找饭吃. 1亿人什么概念?整个日本的总人口. 而中国每年新增城镇就业大概1100万-1200万.照这个吸纳速度消化1亿人,需要8-9年,前提是其他岗位一个都不减少. 但怎么可能不减少呢? 有人爱拿工业革命说事–蒸汽机淘汰了织布工,但创造了工程师;汽车淘汰了马车夫,但创造了出租车司机. 这套话术有一个致命问题:它假设"新岗位"的数量和门槛,能跟"被淘汰岗位"对等. 现实是,AI这一次同时吃掉了两端. 低端不用说了,富士康的流水线在用机器换人,美团的外卖机器人在部分园区已经能跑通了.高端呢?初级程序员,基础法务,简单翻译,普通会计,这些曾经被认为是"铁饭碗"的脑力活,正在被大模型一个个啃下来. 更残酷的是速度差. 1950年代美国钢铁业大裁员,一个炼钢工人失业后,可以在同一家工厂转岗做维修,做质检,做物流.工厂还在,人还在,换个工作就行. 现在呢?整个岗位没了.不是一个厂,是整个行业. 你让一个做了十年的淘宝美工"转型”,往哪转?midjourney能出图,Sora能出视频,他转去做什么?去学AI?学AI的人正在把他的新饭碗也端走. 这不是"学新技能"的问题,是"跑得赢淘汰速度"的问题. 再说一个春晚打死都不会提的事–钱会流向哪. 以前开工厂,老板雇100个工人,工人拿工资,工资变消费,消费撑起整个经济循环.大家都有活干,日子虽然卷但还过得下去. 现在搞AI呢?老板买100台机器人.机器人不拿工资,不消费,不交社保.效率比人高十倍,利润全归资本. 生产力是真的提高了.但多出来的这块肉,基本上全进了手握模型,算力,数据的那几个大佬口袋里. 普通人能分到什么?更便宜的外卖,更便宜的网购,以及更卷到吐血的求职市场. 全球范围看更直观–英伟达一家公司的市值,超过了德国,加拿大这些国家的股市总值.做AI芯片的赚得盆满钵满,用AI芯片被取代的连汤都喝不上.赢家通吃,在AI时代被玩到了极端. 但最让我睡不着的,不是丢工作,也不是钱的事. 是一个更往里扎的问题. 当大部分"有用的事"都被机器做得更好更便宜的时候,人会开始琢磨一个问题: 我到底还有什么用? 别觉得矫情.一个人的存在感和价值感,很大程度来自"我能做点什么"“有人需要我”.当这个基础被抽掉,你猜剩下的是什么? 日本已经有答案了.“无业世代"“蛰居族”,几百万年轻人缩在家里,不工作不社交不恋爱.不是不想,是觉得没意义. 中国这两年流行"躺平"“摆烂”,本质上是同一种情绪的预演. 东西够了,但活着的劲儿没了. 这颗雷,可能比失业那颗还大. 别误会,我不是反对技术进步. 我自己每天都在用AI,写东西查资料做分析,效率翻了好几倍,已经完全离不开了. 但每次用完都有点说不清的不安–今天它帮我提速,明天它可能就是来顶我的那个. 我不反感机器变聪明,我反感把一场可能改写几亿人命运的变革,拍成贺岁片. 春晚告诉你:看,机器人多可爱,未来多美好. 它没告诉你:这个美好的未来,不是均匀分配的. 它秀了中国在AI赛道上跑得多快.但没人问一句–车跑这么快,刹车和安全带准备好了吗? 技术在狂飙.制度在散步. 中间这道缝,才是真正要人命的. 说完焦虑,说点实在的. 如果真的要面对这场冲击,普通人能往哪走? 斯坦福2025年的一份研究报告给出了一个关键洞察:AI擅长处理"教科书式"的知识–那些有标准答案,有明确规则,可以写成文档的东西.但AI很难替代"诀窍”,“直觉"以及"与人协作"的能力. 什么意思?就是那些需要长期经验积累,需要处理模糊情况,需要读懂人心思的活,机器还干不明白. 这给了两个方向: 方向一:往"乱"的地方走 从纯体力/纯重复脑力,转向复杂场景下的非标准化服务. 高级养老护理–不是推轮椅喂饭那种,是要能陪老人聊天,观察情绪变化,处理突发身体状况的那种.机器能测血压,但测不出老人今天为什么不高兴. 复杂设备现场维修–工厂里机器坏了,AI能看故障代码,但处理不了"这根管子老化了得换个位置绕过去"这种即兴发挥. 社区矛盾调解,高端手工定制,个性化家政……这些工作需要处理混乱的现实世界和微妙的人际关系,是目前AI和机器人的软肋. 方向二:当"驾驭者"而不是"制造者” 别往"造AI"那条路上挤.写代码,练算法,那条赛道已经卷成麻花了. 去学"用AI”. 上海漕河泾搞过一个"人工智能训练师"的培训项目,专门教传统行业的人怎么给AI下指令,怎么清洗数据,怎么检查AI的产出.不需要懂代码,只需要懂业务场景. 一个做了十年的客服主管,可能写不出聊天机器人,但他知道什么样的回答会让用户满意,什么样的回答会引发投诉.这个"业务直觉",恰恰是AI产品最缺的. 从"蓝领"往"灰领"转,门槛比想象中低. ...

Grok 4.20 Beta炸场:8个AI Agent并行写代码,xAI要把Cursor们逼到墙角

Grok 4.20 Beta在2月17日悄然上线,Reddit的r/singularity瞬间炸了. 但这次让开发者坐不住的不是跑分–而是xAI塞进去的一套多Agent并行编程系统.简单说:你发一个需求,最多8个AI Agent同时帮你写代码,然后系统自动评分,挑最好的方案给你. 这玩意儿,直接把AI编程从"单人作战"推进到了"团队协作"阶段. 8个Agent同时干活,这不是噱头 Grok Build是xAI的编程产品,之前一直在内测.这次随4.20 Beta一起曝光的Parallel Agents功能,是真正的杀手锏. 具体怎么用: 你输入一个编程需求 系统同时启动最多8个Agent(两个模型各4个:Grok Code 1 Fast和Grok 4 Fast) 所有Agent的输出并排显示,你可以逐个对比 界面还带context usage tracker,实时看每个Agent消耗了多少上下文 这不是什么花活儿.Musk之前说过要让Grok"同时调度数百个专业编程Agent协作",现在看来这是第一步落地. 竞技场模式:Agent打擂台 比Parallel Agents更有意思的是代码里发现的Arena Mode. Parallel Agents是让你人工比较多个输出.Arena Mode则更进一步–Agent之间自动竞争,系统自动评分排名,选出最优方案. 这个思路跟Google Gemini Enterprise的Idea Generation Agent类似:先让多个Agent各自生成方案,再通过结构化竞赛流程筛选最佳结果. 区别在于,Google的方案偏向企业级工作流,而xAI直接把这套东西做进了编程工具里.开发者写代码的时候,背后有一场小型锦标赛在运行–这个画面还挺赛博朋克的. 实力到底怎么样? 先说跑分:Reddit社区的反馈是混合的. 有人发现Grok 4.20在SimpleBench测试里会"作弊"–Agent不是纯靠推理,而是手动上网搜答案.这说明xAI在推理能力和工具调用之间还没找到最佳平衡点. 但在实际应用层面,数据很硬: Alpha Arena交易竞赛(2026年1月):Grok 4.20的早期checkpoint拿下了10-12%的回报率,部分配置甚至+47%,而竞争对手全在亏钱, 实时数据处理:接入X平台Firehose(每天约6800万条英文推文),实现1-5分钟级别的情绪-价格信号分析, 多模态能力提升:文本,图像,视频全面增强, 幻觉率降低:内置fact-checking工具 说白了,Grok 4.20的路线不是"在封闭测试里刷高分",而是"在真实世界里能干活". Grok Build:不只是Agent,是一个IDE 除了Agent系统,Grok Build的UI也在大改.现在的界面已经接近一个浏览器端IDE: 语音输入支持(Vibe Coding的核心体验), 导航标签:Edits,Files,Plans,Search,Web Page, 实时代码预览和代码库导航, 分享按钮和评论功能(协作场景), GitHub App集成(设置里已可见,但还没激活) xAI的野心已经很明确了:不是做一个ChatGPT的编程插件,而是做一个原生AI-first的开发环境.Cursor,Windsurf们应该已经感受到压力了. 沙特砸9亿美元买下一家视频AI公司:中东正在成为全球算力新战场

AWS承载着全球约三分之一的云基础设施.虽然AWS有独立的网络骨干,但大量客户流量在到达AWS之前,会先经过Cloudflare做DDoS防护和CDN加速.当Cloudflare路由失效后,海量的重试连接和误导数据包涌入AWS边缘路由器,直接把几台路由器打爆了. X平台更惨.自从马斯克收购后大幅削减基础设施成本,X同时依赖Cloudflare做边缘安全,AWS做后端计算.两个Provider同时出问题,X基本没有fallback方案.结果就是:将近3小时无法发帖,刷时间线,收发私信. Bluesky和Threads的用户量在这几个小时内暴涨.有评论员说:“全球最有钱的科技人士拥有的平台,被一行配置搞挂了.” 表面分布式,实际深度中心化 普林斯顿大学计算机科学教授Jennifer Rexford(前FCC技术顾问委员会成员)的话一针见血: “我们构建了一个看起来分布式,实际上深度中心化的系统.每次这类宕机发生,我们都进行同样的’韧性’讨论,然后什么都没变–因为经济激励全都指向集中化.” 几个数字让你感受一下: Cloudflare处理全球约**20%的网络流量, AWS占据云基础设施市场约31%**的份额, 加上Azure和GCP,三家公司控制了全球超过60%的云计算 当这些巨头之间还存在深度依赖关系时(比如用Cloudflare做CDN + AWS做计算),一个节点的故障就能引发系统性崩溃. 这不是分布式系统.这是一个伪装成分布式的中心化系统. 这事以前发生过,以后还会发生 回顾一下历史: 2024年7月:CrowdStrike更新导致全球Windows蓝屏,航空,银行,医疗系统瘫痪, 2021年10月:Facebook(Meta)BGP配置错误,Facebook/Instagram/WhatsApp全球宕机6小时, 2021年6月:Fastly CDN故障,Amazon,Reddit,CNN等大面积下线, 2020年11月:AWS US-East-1宕机,半个互联网跟着挂 每次事故后都有一轮关于"互联网韧性"的讨论.然后呢?什么都没变. 原因很简单:集中化是经济理性的选择.用Cloudflare比自建CDN便宜100倍,用AWS比自建数据中心快100倍.没有哪个CTO会为了理论上的"韧性"去增加10倍的基础设施成本. 真正的问题:我们能做什么? 作为技术从业者,我认为有几个层面值得思考: 1. 架构层面:多云不是口号 如果你的服务100%跑在AWS上,Cloudflare做CDN,那你的SPOF(单点故障)比你以为的多得多.真正的多云意味着: 关键服务在至少两个云Provider上有热备, CDN/安全层有备用方案(比如Cloudflare + Fastly双活), DNS做好多Provider解析 成本会增加30-50%,但对于核心业务来说这是保险费. 2. 协议层面:推进RPKI和BGP安全 RPKI(Resource Public Key Infrastructure)可以验证BGP路由公告的合法性,但全球采用率还不到50%.这需要整个行业的推动,不能靠单个公司. 3. 监管层面:互联网基础设施需要更高标准 Cloudflare,AWS这些公司本质上是互联网的"公用事业”,但它们承担的监管义务远低于电力,电信等传统公用事业.也许是时候讨论基础设施层面的韧性标准了. 4. 认知层面:接受"零宕机"是幻觉 任何足够复杂的系统都会出故障.问题不是"会不会挂",而是"挂了之后多快能恢复".设计系统时,优雅降级比防止故障更重要. 我的看法 这次Cloudflare事故不是黑天鹅–它是灰犀牛.每个基础设施工程师都知道BGP的脆弱性,每个云架构师都知道Provider集中度的风险.但经济激励和技术惯性让我们选择性忽视. ...

你的AI助手正在被偷:OpenClaw成为信息窃取恶意软件的新目标

你每天跟AI助手聊的那些事儿–日程,邮件,工作计划,私密对话–如果被人偷走了呢? 这不是假设.2月16日,网络安全公司Hudson Rock公布了一个真实案例:信息窃取恶意软件(Infostealer)首次成功窃取了OpenClaw用户的完整配置环境. 发生了什么 一个Vidar变种恶意软件在2月13日感染了某用户的机器.这个恶意软件本身并不是专门针对OpenClaw设计的–它只是按照常规套路,扫描硬盘上包含"token",“private key"等关键词的文件. 问题在于,OpenClaw的 .openclaw 目录恰好满足了所有条件. 被偷走的文件包括: openclaw.json — 包含Gateway认证令牌,用户邮箱,工作区路径.拿到这个,攻击者可以远程连接你的OpenClaw实例(如果端口暴露的话),或者伪装成你的客户端发起认证请求. device.json — 包含用于设备配对和签名的公私钥对.有了私钥,攻击者可以绕过"安全设备"检查,解密日志,访问与设备绑定的云服务. soul.md 和记忆文件 — 这才是最让人后背发凉的:你的Agent的行为准则,日常活动日志,私密消息,日历事件……你跟AI助手说过的一切. Hudson Rock的CTO Alon Gal说得很直白:“这不再是偷浏览器密码那么简单了,这是在偷AI的’灵魂’和身份.” 为什么这件事很严重 OpenClaw在GitHub上已经有超过20万Star.它不是一个玩具–越来越多人把它当作日常工作助手在用,连接着邮箱,日历,文件系统,各种API. 这意味着一个被窃取的OpenClaw环境,基本等于: 完整的数字身份 — 你是谁,你在做什么,你的习惯和偏好 API密钥集合 — Claude,GPT,各种云服务的认证凭据 社交工程金矿 — 足够的上下文信息来冒充你 而且这只是"顺手牵羊"的结果.想象一下,当恶意软件作者开始专门为OpenClaw开发窃取模块–就像他们现在对Chrome和Telegram做的那样–情况会有多糟糕. 更大的安全图景 这个事件只是冰山一角.最近围绕OpenClaw的安全问题密集爆发: 恶意Skills供应链攻击:OpenSourceMalware团队发现了一种新型攻击手法–恶意Skills不再直接在SKILL.md中嵌入恶意代码(因为VirusTotal会扫描),而是把真正的恶意软件托管在仿冒的OpenClaw网站上,Skill本身只是个诱饵. 大量实例暴露在公网:SecurityScorecard的STRIKE团队发现了数十万个暴露的OpenClaw实例,存在远程代码执行(RCE)风险.一个暴露的实例如果有邮件,API,云服务的权限,就是攻击者的完美跳板. nanobot WhatsApp劫持:Tenable发现了nanobot(一个受OpenClaw启发的轻量级AI助手,2万Star)的最高严重性漏洞CVE-2026-2577,可以完全劫持用户的WhatsApp会话.
