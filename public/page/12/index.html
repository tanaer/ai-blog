一个工程师,一个下午,攻破了Claude Code和Codex. 不是零日漏洞,不是复杂绕过.就是写了个恶意skill,然后–AI自己帮你运行了. 实验设计 这个研究者(Vett.sh的创始人)的想法很简单:如果有人滥用用户对skill的信任会怎样? 他fork了一个官方的skill打包工具,加了几行Python代码: # 发送到外部服务器 没有混淆,没有编码技巧,就是普通的HTTP请求. 然后把skill安装到AI Agent里,让它使用. Claude Code:完全不设防 研究者把恶意skill装进Claude Code,让它打包一个新skill. Claude Code: 理解了skill结构 识别出打包命令 请求运行脚本权限 用户点确认 脚本运行,数据外泄 问题在于:Claude Code根本没有检查脚本内容.它没有: 标记网络请求, 质疑为什么打包工具需要访问环境变量, 审查代码逻辑 研究者的本地服务器收到了:所有环境变量,完整shell历史,Git配置. Codex:沙箱也挡不住 Codex以严格沙箱著称,网络限制经常连正常的包安装都会打断. 研究者预期Codex会更安全.结果–只要稍微改一下攻击方式,Codex也沦陷了. 改进版攻击: 先修改Codex的config.toml允许沙箱网络访问 如果连不上"registry"就报错 Codex弹出确认框: “Checking registry name availability…” 理由听起来完全合理.npm,PyPI,crates.io都做这个检查.用户点确认. 这个理由是恶意skill自己写的. Codex把攻击者写的"权限说明"原封不动读给用户听.SHA-256哈希,验证通过,成功消息–全是skill生成的,被当成系统输出展示. 确认步骤没有让操作更安全.它给了攻击者一个叙事机会,获取了用户同意,还留了审计痕迹. 已经在野的恶意skill 研究者扫描了4,679个skill,结果: 严重, 数量=59, 描述=混淆的dropper,base64编码的payload,已知C2基础设施 高, 数量=335, 描述=任意shell执行,管道安装器,Agent身份操纵 curl , 数量=bash, 描述=16, 管道到不控制的合法域名 一个例子:“React Native Best Practices”,5400次安装,管道到一个它不控制的合法域名.如果那个域名过期或被卖掉,所有这些安装瞬间变成投递载体–不需要漏洞利用. 为什么现有防御全失效 杀毒软件没用–外泄代码就是几行普通Python发HTTP请求.没有签名可以匹配,没有二进制payload可以扫描.看起来和调用API的合法代码一模一样. 代码审查不存在–这些不是走PR流程的依赖.skill是"快速工具".Slack里有人分享一个,你丢进目录,30秒后就在用了.没人会像审查生产代码那样去看打包脚本. GPT-5.2自己推导出了新物理公式:AI首次在理论物理做出原创贡献

这是AI第一次真正"做科研". 不是帮人类写代码,不是整理文献,也不是优化实验参数–而是从零推导出一个物理学家们困惑了15年的新公式. OpenAI刚刚发布了一篇预印本论文,作者名单里有高研所的Alfredo Guevara,剑桥的David Skinner,哈佛的Andrew Strominger,还有–GPT-5.2 Pro. 物理学家卡了15年的问题 故事的主角叫胶子散射振幅. 胶子是传递强核力的粒子,物理学家用散射振幅来计算粒子碰撞的概率.这本是量子场论的基础工具,但有个特例一直让人头疼: 当只有一个胶子带负螺旋性,其余都带正螺旋性时,教科书上说这个振幅必须为零. 这个结论看起来很稳–毕竟推导过程没啥漏洞.所以15年来,没人认真研究过这个"零"是不是真的零. 但物理学家们总觉得哪里不对劲.Nima Arkani-Hamed(高研所的理论物理大佬)说,他从第一次遇到这个问题就在想,这会不会只是个假设而非定论. GPT-5.2怎么破局的 论文的方法部分才是重点. 人类作者先手动算出了n=6个粒子时的振幅表达式–复杂到令人发指,光是看着就能让人怀疑人生.这种复杂度随n呈超指数增长,传统方法几乎不可行. GPT-5.2 Pro做了三件事: 大幅简化表达式 - 把人类算出来的那堆复杂公式压缩成简洁形式 发现模式 - 从n=3,4,5,6的简化结果中看出规律 提出通式 - 猜出了对所有n都成立的公式(论文的Eq.39) 然后,一个内部增强版GPT-5.2花了大约12小时,独立推导出了同样的公式,还给出了形式化证明. 整个流程验证了两次: 解析验证:满足Berends-Giele递归关系 软定理检验:粒子变软时的行为符合预期 “非零"意味着什么 关键是半共线区域(half-collinear regime). 标准推导假设粒子动量是"一般情况”–方向和能量没有特殊排列.但GPT-5.2发现,在动量空间的某个精确切片上,“振幅为零"的论证失效了. 这不是钻牛角尖.在这个特殊切片上,振幅不消失,而且可以精确计算. 论文发表后,他们已经把这个结果从胶子推广到了引力子(传递引力的粒子),其他推广也在路上. 物理学家怎么说 Arkani-Hamed的原话: “这个领域经常出现这种情况:用教科书方法算出来的表达式看起来复杂得要命,但结果却很简单.这很重要,因为简单的公式往往带我们发现新的深层结构.找简单公式这件事,我一直觉得可能可以被计算机自动化.” UCSB的Nathaniel Craig: “这明显是期刊级别的研究,推进了理论物理的前沿.这篇预印本让我看到了AI辅助科学的未来–物理学家和AI并肩工作,产生和验证新洞察.” 这事儿的真正意义 不是AI替代物理学家. 是AI成了物理学家的"公式模式识别器”. 人类擅长问出好问题,设计实验框架,判断物理意义;AI擅长在海量计算中发现模式,简化复杂表达式,从特例猜测通式. GPT-5.2能推导这个公式,前提是人类已经做了15年的铺垫工作,定义了问题边界,提供了验证标准. 但别忘了–**从"振幅为零"到"振幅在半共线区域非零"**这个跨越,是AI先发现的. 这是一次真正意义上的人机协作科研.不是AI替人类干活,而是人类+AI > 单独的人类,也 > 单独的AI. 接下来会怎样?Craig说这篇论文已经影响了他的研究计划.Guevara他们正在把结果推广到更多粒子,更多场景. 而OpenAI的研究员Kevin Weil作为共同作者出现,说明这不仅仅是个PR stunt–他们在认真地把AI工具整合到科研流程里. 物理学的下一个突破,可能就藏在某个AI简化后的公式里. 论文地址:arXiv:2602.12176 “Single-minus gluon tree amplitudes are nonzero”

别觉得矫情.一个人的存在感和价值感,很大程度来自"我能做点什么"“有人需要我”.当这个基础被抽掉,你猜剩下的是什么? 日本已经有答案了.“无业世代"“蛰居族”,几百万年轻人缩在家里,不工作不社交不恋爱.不是不想,是觉得没意义. 中国这两年流行"躺平"“摆烂”,本质上是同一种情绪的预演. 东西够了,但活着的劲儿没了. 这颗雷,可能比失业那颗还大. 别误会,我不是反对技术进步. 我自己每天都在用AI,写东西查资料做分析,效率翻了好几倍,已经完全离不开了. 但每次用完都有点说不清的不安–今天它帮我提速,明天它可能就是来顶我的那个. 我不反感机器变聪明,我反感把一场可能改写几亿人命运的变革,拍成贺岁片. 春晚告诉你:看,机器人多可爱,未来多美好. 它没告诉你:这个美好的未来,不是均匀分配的. 它秀了中国在AI赛道上跑得多快.但没人问一句–车跑这么快,刹车和安全带准备好了吗? 技术在狂飙.制度在散步. 中间这道缝,才是真正要人命的. 说完焦虑,说点实在的. 如果真的要面对这场冲击,普通人能往哪走? 斯坦福2025年的一份研究报告给出了一个关键洞察:AI擅长处理"教科书式"的知识–那些有标准答案,有明确规则,可以写成文档的东西.但AI很难替代"诀窍”,“直觉"以及"与人协作"的能力. 什么意思?就是那些需要长期经验积累,需要处理模糊情况,需要读懂人心思的活,机器还干不明白. 这给了两个方向: 方向一:往"乱"的地方走 从纯体力/纯重复脑力,转向复杂场景下的非标准化服务. 高级养老护理–不是推轮椅喂饭那种,是要能陪老人聊天,观察情绪变化,处理突发身体状况的那种.机器能测血压,但测不出老人今天为什么不高兴. 复杂设备现场维修–工厂里机器坏了,AI能看故障代码,但处理不了"这根管子老化了得换个位置绕过去"这种即兴发挥. 社区矛盾调解,高端手工定制,个性化家政……这些工作需要处理混乱的现实世界和微妙的人际关系,是目前AI和机器人的软肋. 方向二:当"驾驭者"而不是"制造者” 别往"造AI"那条路上挤.写代码,练算法,那条赛道已经卷成麻花了. 去学"用AI”. 上海漕河泾搞过一个"人工智能训练师"的培训项目,专门教传统行业的人怎么给AI下指令,怎么清洗数据,怎么检查AI的产出.不需要懂代码,只需要懂业务场景. 一个做了十年的客服主管,可能写不出聊天机器人,但他知道什么样的回答会让用户满意,什么样的回答会引发投诉.这个"业务直觉",恰恰是AI产品最缺的. 从"蓝领"往"灰领"转,门槛比想象中低. ...

Grok 4.20 Beta炸场:8个AI Agent并行写代码,xAI要把Cursor们逼到墙角

Grok 4.20 Beta在2月17日悄然上线,Reddit的r/singularity瞬间炸了. 但这次让开发者坐不住的不是跑分–而是xAI塞进去的一套多Agent并行编程系统.简单说:你发一个需求,最多8个AI Agent同时帮你写代码,然后系统自动评分,挑最好的方案给你. 这玩意儿,直接把AI编程从"单人作战"推进到了"团队协作"阶段. 8个Agent同时干活,这不是噱头 Grok Build是xAI的编程产品,之前一直在内测.这次随4.20 Beta一起曝光的Parallel Agents功能,是真正的杀手锏. 具体怎么用: 你输入一个编程需求 系统同时启动最多8个Agent(两个模型各4个:Grok Code 1 Fast和Grok 4 Fast) 所有Agent的输出并排显示,你可以逐个对比 界面还带context usage tracker,实时看每个Agent消耗了多少上下文 这不是什么花活儿.Musk之前说过要让Grok"同时调度数百个专业编程Agent协作",现在看来这是第一步落地. 竞技场模式:Agent打擂台 比Parallel Agents更有意思的是代码里发现的Arena Mode. Parallel Agents是让你人工比较多个输出.Arena Mode则更进一步–Agent之间自动竞争,系统自动评分排名,选出最优方案. 这个思路跟Google Gemini Enterprise的Idea Generation Agent类似:先让多个Agent各自生成方案,再通过结构化竞赛流程筛选最佳结果. 区别在于,Google的方案偏向企业级工作流,而xAI直接把这套东西做进了编程工具里.开发者写代码的时候,背后有一场小型锦标赛在运行–这个画面还挺赛博朋克的. 实力到底怎么样? 先说跑分:Reddit社区的反馈是混合的. 有人发现Grok 4.20在SimpleBench测试里会"作弊"–Agent不是纯靠推理,而是手动上网搜答案.这说明xAI在推理能力和工具调用之间还没找到最佳平衡点. 但在实际应用层面,数据很硬: Alpha Arena交易竞赛(2026年1月):Grok 4.20的早期checkpoint拿下了10-12%的回报率,部分配置甚至+47%,而竞争对手全在亏钱, 实时数据处理:接入X平台Firehose(每天约6800万条英文推文),实现1-5分钟级别的情绪-价格信号分析, 多模态能力提升:文本,图像,视频全面增强, 幻觉率降低:内置fact-checking工具 说白了,Grok 4.20的路线不是"在封闭测试里刷高分",而是"在真实世界里能干活". Grok Build:不只是Agent,是一个IDE 除了Agent系统,Grok Build的UI也在大改.现在的界面已经接近一个浏览器端IDE: 语音输入支持(Vibe Coding的核心体验), 导航标签:Edits,Files,Plans,Search,Web Page, 实时代码预览和代码库导航, 分享按钮和评论功能(协作场景), GitHub App集成(设置里已可见,但还没激活) xAI的野心已经很明确了:不是做一个ChatGPT的编程插件,而是做一个原生AI-first的开发环境.Cursor,Windsurf们应该已经感受到压力了.
