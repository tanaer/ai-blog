传统的编程基准测试很简单:运行代码,检查输出,通过或失败.

但 Frontier-CS 不一样.它的 240 个问题是**开放式,连续评分**的:

- 优化一个 CUDA kernel 让它尽可能快
- 设计一个缓存淘汰策略让 miss rate 最小
- 实现一个跨核并行的排序算法

没有"正确答案",只有从"编译失败"到"超越参考实现"的光谱.

**问题是:评估比生成本身更难.**

## 四大挑战

### 1. 连续评估,不是一次性的
学术基准通常发一篇论文就完了.Frontier-CS 是**活着的排行榜**.

过去几个月:Gemini 3.0 Pro,GPT 5.2 Thinking,Grok 4,DeepSeek 3.2......每个新模型都要跑 240 道题.OpenAI 刚发了 GPT-5.3-codex,Anthropic 就发了 Claude Opus 4.6.

**解决方案**:基于 hash 的断点续跑.每个结果记录 solution 文件和 problem 目录的 SHA hash.恢复时自动重跑变更的,保留未变的.

### 2. 异构硬件规模化
一次批量跑可能包含:
- 需要 A100 GPU 的 CUDA 优化题
- 需要 c7i.4xlarge(16 vCPU)的 HPC 负载
- 只需要普通 CPU 的算法题

**解决方案**:ResourceSignature 分组.把题目按 (cloud, accelerators, instance type) 分类,每组分配专属的 SkyPilot 集群池.CPU-only 跑便宜实例,GPU 题跑 GPU 实例.

### 3. 评估确定性
换 A100 为 H100?排行榜所有数字都变了.

RL 社区已经吃过苦头:MuJoCo 和 Gym 版本升级,几年发表的 baseline 全废了.

**解决方案**:
- 锁定一切:硬件规格,CUDA 版本,运行时环境
- Docker 镜像 + AMI 固化 OS,驱动,库
- 每个结果打上 solution hash + problem hash
- 任何变化都自动失效旧分数,触发重跑

不求跨环境可移植,只保证同一环境下的可复现.

### 4. 生成 ≠ 评估
很多基准把生成和评估耦合在一起:调 API,拿输出,打分.

Frontier-CS **彻底解耦**:
- `gen/` 模块调用 LLM,写入源文件, 评估管线只看源文件,不关心谁写的,什么 prompt,试了几次, 每次评估在隔离容器/VM 里跑,看不到其他方案,读不到测试用例,没网

**结果**:手写方案和 LLM 方案用同一套评估路径.可以公平对比不同 prompting 策略,不同温度,不同模型.

## 未来挑战:Agentic Submissions
现在提交的是源文件.但趋势是 agentic workflow:模型读题,写代码,跑起来,看输出,迭代.

这带来两个新问题:

**1. Reward Hacking**

Agent 迭代打分 harness 容易过拟合.发现缓存 trace 的时序局部性并设计策略 → 学到了真东西.背下测试集访问模式硬编码查表 → 什么都没学到.两种都高分,只有一种泛化.

**2. 成本模型崩塌**

现在 API 调用(生成)和 GPU 计算(评估)是分开的.Agentic 提交需要:写代码 → GPU 跑 → 看结果 → 再写.GPU 集群得一直开着等 agent 思考.两个昂贵资源同时烧,一个等另一个.这是 API 延迟和 GPU 利用率的联合调度问题,目前没有现成解法.

## 实际应用场景
**场景一:模型能力深度评估**

SWE-bench 告诉你模型能不能修 bug.Frontier-CS 告诉你模型能不能优化 CUDA kernel,设计分布式系统,做 HPC 编程.对基础设施团队更有参考价值.

**场景二:自建评估系统参考**

你要评估 LLM 在自有任务上的表现?Frontier-CS 的架构(hash-based resume,resource grouping,sandbox isolation)可以直接复用.

**一句话**:评估 LLM 比让 LLM 生成代码更难.Frontier-CS 展示了如何构建一个可信,可复现,可扩展的评估系统.
