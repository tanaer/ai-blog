## 一句话总结
**Cord 让 AI Agent 自己决定如何分解任务**--不是预定义工作流,而是 Agent 在运行时动态构建任务树,自动管理依赖,并行执行和上下文传递.

## 现有框架的问题
多 Agent 框架正在爆发,但它们都在解决错误的问题:

**LangGraph**, 方式=状态机,预定义节点和边, 问题=图是静态的,Agent 无法中途改变分解方式
**CrewAI**, 方式=角色扮演(研究员,分析师,作家), 问题=角色由开发者预设,Agent 不能动态调整团队
**AutoGen**, 方式=群聊模式, 问题=无结构,无依赖追踪,协调从对话中涌现,难以预测
**OpenAI Swarm**, 方式=轻量级 Agent 切换, 问题=线性执行,无并行,无树结构
**Claude Tool-use**, 方式=单 Agent 循环, 问题=上下文窗口限制,无法并行

**共同问题**:所有框架都要求开发者**预定义协调结构**.你决定工作流图,Agent 角色,切换模式.Agent 在你的边界内执行.

但现在的模型(Claude,GPT-4)已经很擅长规划.它们能自然地将问题分解为子问题,理解依赖关系,知道什么时候任务太大需要拆分.

**那为什么还要硬编码分解逻辑?**

## Cord 的解法:让 Agent 构建树
Cord 的核心洞察:**给 Agent 一个目标,让它自己决定如何分解**.

### 使用示例
```bash
cord run "Should we migrate our API from REST to GraphQL? Evaluate and recommend."
```

一个 Agent 启动,读取目标,决定需要研究才能回答,然后创建子任务:

● #1 [active] GOAL Should we migrate our API from REST to GraphQL?
 ● #2 [active] SPAWN Audit current REST API surface
 ● #3 [active] SPAWN Research GraphQL trade-offs for our stack
 ○ #4 [pending] ASK How many concurrent users do you serve?
 blocked-by: #2
 ○ #5 [pending] FORK Comparative analysis
 blocked-by: #3, #4
 ○ #6 [pending] SPAWN Write migration recommendation
 blocked-by: #5

**没有工作流被硬编码**.Agent 在运行时决定了这个结构:
- 并行化 API 审计 (#2) 和 GraphQL 研究 (#3)
- 创建询问节点 (#4),因为推荐取决于规模(无法自行研究)
- 将 #4 阻塞在 #2 之后,因为问题在审计结果作为上下文时更有意义
- 将 #5 设为 fork,使分析继承所有已学内容
- 将最终推荐排序在分析之后

## 核心创新:Spawn vs Fork
这是 Cord 真正新颖的地方:**spawn 和 fork 作为上下文流原语**.

### Spawn(生成)
```python
spawn(goal, prompt, blocked_by)

- 子 Agent 获得**干净的状态**, 只有它的提示和显式依赖节点的结果, 像雇佣承包商--这是规格,去吧, 便宜,易重启,易推理

### Fork(分叉)
fork(goal, prompt, blocked_by)

- 子 Agent 获得**所有已完成兄弟节点的结果注入上下文**, 像向团队成员简报--他们知道团队学到的所有内容, 更昂贵,但对于需要基于先前工作的分析是必要的

**关键区别**:不是关于并发(两者都可以并行或顺序运行),而是关于**子节点知道什么**.

在上面的例子中,Agent 选择:
- **spawn** 用于独立研究任务
- **fork** 用于需要所有内容的分析

Agent 自己做出了这个选择--模型直观理解这个区别.

## 技术实现
每个 Agent 是一个 Claude Code CLI 进程,MCP 工具由共享的 SQLite 数据库支持:

- `fork(goal, prompt, blocked_by)`: 创建继承上下文的子任务
- `ask(question, options)`: 向人类提问, `complete(result)`: 标记自己完成, `read_tree()`: 查看完整协调树

Agent 不知道自己在协调树中.它们看到工具并根据需要使用.协议--依赖解析,权限范围,结果注入--由 MCP 服务器强制执行.

当询问节点就绪时,引擎暂停以在终端提示人类.答案存储为结果,下游节点解除阻塞.**人类是树的参与者,不是观察者**.

**代码量**:~500 行 Python.SQLite + MCP.

# 克隆仓库
git clone https://github.com/kimjune01/cord.git
cd cord

# 安装依赖
uv sync

# 运行任务
cord run "your goal here" --budget 2.0

# 或指向规划文档
cord run plan.md --budget 5.0

根 Agent 读取 markdown 并将其分解为协调树.你可以用任何方式写计划--要点,章节,散文--Agent 会找出任务结构,依赖和并行性.

**要求**:Claude Code CLI 和包含它的订阅.

## 为什么这很重要

### 对开发者
不再需要预定义工作流.你只需要:
1. 给 Agent 一个目标
2. 让它决定如何分解
3. 在关键节点介入(ask)

### 对 Agent
- 能根据任务复杂度动态调整策略, 能并行化独立子任务, 能在需要时向人类提问, 能继承或隔离上下文

### 对未来
Cord 的协议(五个原语,依赖解析,权限范围,两阶段生命周期)独立于具体实现.你可以:
- 用 Postgres 实现多机器协调
- 直接用 Claude API,无 CLI 开销
- 多 LLM 提供商--GPT 处理便宜任务,Claude 处理复杂任务
- 某些节点用人类工作者

## 写在最后
Cord 代表了一种新的多 Agent 协调范式:**从预定义工作流转向动态任务树**.

当模型足够智能,能够自己规划时,硬编码协调结构就像给赛车手规定路线--限制了它们的能力.

让 Agent 自己决定如何分解任务,可能是多 Agent 系统的下一个进化方向.
