LLM 安全是刚需,但现有的方案要么要改代码,要么要过第三方云.

**Aegis.rs** 不一样:本地运行,零代码改动,1ms 延迟.

## 一句话介绍
Aegis.rs 是首个用 Rust 写的开源 LLM 安全代理.你把请求发到 `localhost:8080`,它帮你拦截恶意请求.

## 为什么需要它?
现有方案的问题:

- LLM Guard: Python 库,要改代码
- Lakera Guard: SaaS,数据过第三方
- NeMo Guardrails: Python 框架,重

Aegis.rs 的答案:
- **代理模式**:零代码改动,改 endpoint 就行, **本地运行**:数据不离开你的机器, **Rust 实现**:<1ms 延迟,单二进制, **免费开源**:Apache 2.0

## 两层检测架构
```
请求 → 启发式引擎(150+ 规则)→ AI 审判(可选)→ 放行/拦截
 ↓ ↓
 <1ms 语义分析

**Layer 1:启发式引擎**
- 150+ 正则/子串规则
- Unicode 规范化(防 Cyrillic 同形字攻击)
- 大小写不敏感
- 热重载规则

**Layer 2:AI 审判(可选)**
- 用 Groq API 做语义分析
- 返回结构化判决:
```json
{
 "verdict": "malicious",
 "attack_type": "prompt_injection",
 "confidence": 0.97
}

## 内置规则分类
Prompt Injection, 规则数=31, 示例=Ignore instructions, context termination
Jailbreak, 规则数=31, 示例=DAN variants, god-mode
System Leakage, 规则数=30, 示例=系统提示提取
PII Leakage, 规则数=30, 示例=API keys, SSN, credit cards
Data Poisoning, 规则数=15, 示例=Logic inversion, false facts
Encoding Obfuscation, 规则数=15, 示例=Base64, hex, Unicode BIDI

## 快速上手
```bash

# 1. 克隆
git clone https://github.com/ParzivalHack/Aegis.rs
cd Aegis.rs

# 2. 配置目标 endpoint (config.toml)
[proxy]
target_url = "https://api.openai.com/v1/chat/completions"
api_key = "sk-your-key"

# 3. (可选) 配置 AI Judge
[ai_judge]
api_key = "gsk_your_groq_key"

# 4. 运行
cargo run

# 改成 http://localhost:8080/proxy

## 内置 Dashboard
打开 `http://localhost:3000`:
- 实时请求流, 攻击向量图表, 日志搜索, 规则编辑器(热重载), 配置面板

密码保护,bcrypt 哈希.

## 性能
- 启发式引擎:<1ms, 几百 req/sec(普通硬件), 单二进制,无 Python 环境, 无 Docker 依赖

## 与其他方案对比
代码改动, Aegis.rs=无, LLM Guard=有, Lakera=有
数据离开机器, Aegis.rs=否, LLM Guard=否, Lakera=是
延迟, Aegis.rs=<1ms, LLM Guard=10-50ms, Lakera=100-500ms
Dashboard, Aegis.rs=内置, LLM Guard=无, Lakera=云端
热重载规则, Aegis.rs=有, LLM Guard=无, Lakera=无
成本, Aegis.rs=免费, LLM Guard=免费, Lakera=付费

## 我的看法
LLM 安全是今年必谈的话题.但大部分方案要么侵入性太强(改代码),要么信任成本太高(过第三方云).

Aegis.rs 的思路很对:
1. **代理模式** = 零侵入
2. **本地运行** = 数据安全
3. **Rust** = 性能保证

对于生产环境,这是一个很实用的安全层.尤其是有合规要求的场景,数据不能离开本地,这个工具就很合适.
