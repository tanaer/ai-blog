JetBrains 的 Databao Agent 在 Spider 2.0-DBT 基准测试拿了第一.有趣的是,他们的 baseline 最初只能解决 **25%** 的任务--不是靠换模型翻盘,而是**把 AI Agent 当成初级工程师来带**.

## 这个 benchmark 测的是什么?
Spider 2.0-DBT 不是简单的 SQL 生成测试.它给 Agent 一个**不完整的 dbt 项目**,里面有缺失的模型,错误的代码,Agent 需要:

- 读整个仓库,理解项目结构, 找出哪里坏了,哪里缺了, 写 SQL 修复或实现新模型, 运行 `dbt build`,看报错,迭代, 直到项目构建成功

68 个任务,有些项目有 **2500+ 列的表**,**几十个模型**,**数千行 SQL**.这不是"写个 SELECT 语句"能搞定的--Agent 得像真正的数据工程师一样工作.

## 为什么 Databao 能赢?
他们的核心洞察:

> **可靠性胜过聪明,prompt 买不来可靠性--你得为它设计.**

Databao 团队发现,性能提升最大的改动不是换模型,而是:

1. **减少不确定性**:限制 Agent 能访问的工具和数据,避免信息过载
2. **升级上下文**:提供清晰的项目文档和依赖关系,而不是让 Agent 瞎猜
3. **强制工作流**:像 code review 一样约束 Agent 的行为,不让他乱来
4. **清晰的 policy**:把一堆混乱的 prompt 重写成明确的规则

这听起来像什么?**像在带一个 junior 工程师**.

## 这意味着什么?
大部分 AI Agent 项目还在追求"更聪明的模型",但 Databao 的经验表明:**工程方法论才是瓶颈**.

从 25% 到第一名的跨越,不是 prompt engineering 的魔法,而是系统设计的胜利.当你把 Agent 当成需要指导的初级同事--给它清晰的边界,可靠的工具,明确的工作流--它的表现会远超"放养"的 Agent.

这可能是 2026 年 AI Agent 开发的最大教训:**别指望模型自己变聪明,设计让它变聪明.**
