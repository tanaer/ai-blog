刚发现一个有意思的技能改进工具:Sensei

## 谁能用?
写 AI Agent 技能文件的.技能写得不好,Agent 不知道什么时候用,什么时候不用--这叫 **skill collision**.

常见问题:
- 没有 triggers → Agent 不知道何时激活
- 没有 anti-triggers → Agent 不该用时乱用
- 描述太短 → 匹配不准
- Token 太多 → 浪费上下文窗口

Sensei 的做法:**用 Ralph 循环自动改进,直到合规**.

## 怎么用?
```bash

# 安装
git clone https://github.com/spboyer/sensei.git ~/.copilot/skills/sensei
cd ~/.copilot/skills/sensei/scripts && npm install

# 用法
Run sensei on my-skill-name # 单个技能
Run sensei on skill-a, skill-b # 多个技能
Run sensei on all Low-adherence skills # 所有低合规技能
Run sensei on my-skill-name --fast # 快速模式(跳过测试)
```

## Ralph 循环是什么?
Read → Score → Improve → Verify → Check Tokens → Summary → Prompt → Repeat

1. **Read**: 读取 SKILL.md,统计 token
2. **Score**: 运行规则检查(描述长度,触发词,反触发词)
3. **Scaffold**: 如果测试目录不存在,自动创建
4. **Improve**: 添加 USE FOR / DO NOT USE FOR / compatibility
5. **Verify**: 运行测试,失败就修复
6. **Check Tokens**: 验证 token < 500
7. **Summary**: 显示改进前后对比
8. **Prompt**: 问你要不要 Commit / Create Issue / Skip
9. **Repeat**: 直到 Medium-High 或迭代 5 次

## 合规级别
- Low: 只有基础描述
- Medium: 有触发词
- Medium-High: 有 USE FOR + DO NOT USE FOR
- High: Medium-High + 路由清晰(INVOKES/FOR SINGLE OPERATIONS)

## Before/After 示例
**Before (Low)**:
```yaml
name: pdf-processor
description: 'Process PDF files for various tasks'

问题:只有 37 字符,无触发词,无反触发词.

**After (Medium-High)**:

 Process PDF files including text extraction, rotation, and merging.
 USE FOR: "extract PDF text", "rotate PDF", "merge PDFs", "split PDF".
 DO NOT USE FOR: creating new PDFs (use document-creator), extracting
 images (use image-extractor).

改进:~350 字符,清晰触发词,反触发词防止冲突.
