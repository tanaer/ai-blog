你的仪表盘越来越慢,Postgres 快被分析查询压垮了.你知道答案是 **Postgres + ClickHouse**:事务用 Postgres,分析用 ClickHouse.

但迁移?太痛苦了.AI 来帮忙?大多数 AI 辅助迁移产出的都是垃圾.

ClickHouse 官方博客刚分享了一个方法:**给 AI Agent 配一个"Harness"(工具链框架)**,让它能有效完成迁移工作.

## 为什么普通 AI 迁移会失败?
写个玩具 Demo 没问题.真正难的是生产环境:

- 重构数据模型以适应 OLAP 性能, 把读时聚合改成写时物化视图, 优化 API 里的查询, 改完前端组件还要验证

这不是"能不能写 ClickHouse SQL"的问题,而是"能不能在复杂的应用栈里有效工作"的问题.

## 三要素:Code + Cadence + Context

### 1. 一切皆代码(Code)
把整个数据栈当成一个用代码表达的系统--不是散落的 DDL 脚本和 SQL 字符串,而是共享的,带类型的对象:

```typescript
import { MaterializedView, OlapTable, sql } from "@514labs/moose-lib";

// 定义目标表 schema(带类型和注释)
interface TargetSchema {
 id: string;
 /** 所有评分的平均值 */
 average_rating: number;
 /** 有效评论总数 */
 num_reviews: number;
}

// 在代码中声明 ClickHouse 表
export const targetTable = new OlapTable<TargetSchema>("target_table", {
 orderByFields: ["id"],
});

// 物化视图:把查询时聚合变成预计算
export const mvToTargetTable = new MaterializedView<TargetSchema>({
 selectStatement: sql`
 SELECT
 ${sourceTable.columns.id},
 avg(${sourceTable.columns.rating}) AS average_rating,
 count(*) AS num_reviews
 FROM ${sourceTable}
 GROUP BY ${sourceTable.columns.id}
 `,
 selectTables: [sourceTable],
 targetTable,
 materializedViewName: "mv_to_target_table",
```

这样 AI Agent 就在它熟悉的环境(TypeScript + Git)里工作了.

### 2. 快速反馈循环(Cadence)
Boyd 定律:**迭代速度胜过迭代质量**.如果每次改动都要云端部署 + 回填才能验证,AI 迭代太慢就没用了.

MooseStack 提供三层反馈:

- **IDE 层**:写代码时就能发现 schema,类型,SQL 问题
- **本地开发层**(`moose dev`):热切换基础设施,真实数据,运行时错误
- **预览分支层**:云端生产级验证后再合并

### 3. 注入最佳实践(Context)
给 Agent 注入 ClickHouse 的专业知识:
- Agent Skills(技能文件)
- 参考文档链接
- InfraMap(依赖关系图)通过 MCP 暴露给 Agent

当 Agent 修改源 schema 或聚合逻辑时,不用猜什么会坏--依赖关系是明确的,影响范围一目了然.

## 实际效果
从创业公司到大型企业,这套方法把生产环境分析负载从 Postgres 迁移到 ClickHouse 的时间,从**几个月缩短到几天**.

## 应用场景
**场景一:仪表盘性能优化**

你的 SaaS 后台分析页面加载要 10 秒,用这套方法把聚合查询下推到 ClickHouse 物化视图,Agent 可以自动完成 schema 重构,视图创建,API 修改.

**场景二:混合事务分析架构**

你想保持 Postgres 处理订单,但把用户行为分析放到 ClickHouse.Agent Harness 可以帮你在两种数据库间建立数据管道,保持类型一致性.

**一句话**:AI Agent 单干数据迁移容易翻车,但给它配好工具链(Code + Cadence + Context),就能从几个月缩短到几天.
