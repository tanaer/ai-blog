AI Agent 写代码已经很强了,但有一类 bug 它们几乎搞不定:**并发 bug.**

CMU 的 PASTA Lab 刚发布了 **Spaghetti Bench**,专门测试 AI Agent 修复竞态条件(race condition)的能力.测了 6 个顶级模型,结果挺扎心的.

## 为什么并发 Bug 这么难?
SWE-bench 上几百个任务,只有**一个**是竞态条件.这不是巧合--并发 bug 有个致命特点:

**测试不可靠.** 你跑 100 次测试全绿,不代表 bug 修好了.只是那个触发 bug 的线程调度恰好没出现.

AI Agent 通常的验证方式是循环跑测试:

```bash
for i in {1..100}; do java MyProgram; done
```

100 次全过?修好了!但其实可能根本没修好.

## 6 大模型实测结果
CMU 用 [OpenHands](https://docs.openhands.dev/) 框架测试了:

- Claude Opus 4.5, Claude Sonnet 4.5, GPT-5.2
- Qwen 3 Coder 480B
- Gemini 3 Flash Preview

测试分两组:
- **Easy 组**(28 题):单文件 100-200 行的并发小程序
- **Hard 组**(11 题):Apache Kafka 项目中的真实并发 bug

关键对比:**有 Fray 工具 vs 没有 Fray 工具**

[Fray](https://github.com/cmu-pasta/fray) 是 CMU 开发的确定性并发测试工具,能系统地探索所有线程调度组合,几秒内可靠触发 bug.

## 一个真实案例
以 WorkStealQueue(无锁工作窃取队列)为例:

**没有 Fray 时**,Claude Sonnet 4.5 改了一个字符(`<=` → `<`),跑完测试没报错就交了.但 Fray 在第 2,616 次迭代找到了 bug--修复是假的.

**有了 Fray 后**,同一个模型经过 5 轮迭代,产出了真正全面的修复:
- 把 `elems` 数组和 `mask` 改成 volatile
- 重写了 `pop()` 方法,用锁 + 双重检查处理单元素场景
- 在 `push()` 中增加了 resize 竞态检测

Fray 验证跑了 **10 万次迭代,零错误**.

## 核心洞察
> 没有合适的工具,即使最强的模型也会产出"看起来对但实际错"的补丁.

AI Agent 的问题不是"不够聪明",而是**缺少正确的验证手段**.给它一个能确定性重放线程调度的工具,修复率会显著提升.

## 应用场景
**场景一:CI/CD 中的并发测试**

如果你的项目有多线程代码,把 Fray 集成到 CI 流水线,在 AI 生成补丁后自动验证.别再依赖"跑 100 次全绿"了.

# Fray 可以系统探索线程调度
git clone https://github.com/cmu-pasta/fray
cd fray && ./gradlew build

**场景二:评估 AI 编码工具**

SWE-bench 评不出你的 AI Agent 能不能处理并发问题.用 Spaghetti Bench 做补充评测,尤其对后端/基础设施项目至关重要.

**一句话**:AI Agent 修 bug 很强,但修并发 bug 很弱--除非你给它正确的工具.Spaghetti Bench 让这个盲区第一次有了量化标准.
