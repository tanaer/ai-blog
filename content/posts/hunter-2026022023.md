你每天用 Claude Code,ChatGPT,DeepSeek,Grok,Warp......每个会话都是孤立的信息孤岛.上周在 Claude Code 里解决的 Supabase auth 问题,今天又在 ChatGPT 里问了一遍.

**Session-graph** 解决这个问题:把分散的 AI 编码会话转化为**可查询的知识图谱**.

## 核心问题
开发者每天用 5+ 个 AI 工具,每个会话都是:
- 孤立的 silo, 标签页一关,知识就消失, grep 会话日志只能得到文本墙,没有结构

**现有方案的问题**:单平台,平面文件.只能搜索一个工具的历史,不能跨平台关联.grep 不能告诉你 "FastAPI 使用 Pydantic" 或 "Neo4j 是图数据库".

## Session-graph 的解决方案
从所有 AI 编码会话中提取**结构化知识三元组** -- (subject, predicate, object):

```
"How does FastAPI relate to Pydantic?"
→ FastAPI --uses--> Pydantic

"What sessions discussed authentication?"
→ 3 sessions across Claude Code + DeepSeek

### 技术亮点
**1. 多平台支持**
- Claude Code (JSONL), DeepSeek (JSON zip), Grok (JSON), Warp (SQLite)
- ChatGPT / Cursor (计划中)

**2. 正式本体论**
组合 5 个 W3C/ISO 标准,不造轮子:
- PROV-O:来源追溯, SIOC:对话结构, SKOS:分类体系, Dublin Core:元数据, Schema.org:代码实体

**3. Wikidata 链接**
- 实体链接到 1 亿+ Wikidata 条目
- "k8s","kubernetes","K8s" 都解析到 Q22661306
- 跨语言去重:"medication" 和 "medicamento" 都映射到 Q12140

**4. 完整来源追溯**
每个三元组都能追溯到:
- 具体消息内容, 所属会话, 平台来源, 磁盘文件路径

**5. 联邦查询**
SPARQL 的 SERVICE 关键字让你同时查询本地图谱和 Wikidata:

```sparql

# 本地图谱中的实体,Wikidata 的描述
SELECT ?localLabel ?wikidataDescription WHERE {
 ?entity a devkg:Entity ;
 rdfs:label ?localLabel ;
 owl:sameAs ?wd .
 SERVICE <https://query.wikidata.org/sparql> {
 ?wd schema:description ?wikidataDescription .
 FILTER(LANG(?wikidataDescription) = "en")
 }

## 实际数据
52 个真实会话的处理结果:

- Fuseki 中总三元组: 138,802, 索引会话数: 52, 提取知识三元组: ~2,500+, 不同实体: ~4,600+, Wikidata 链接实体: ~33%, 支持平台: 4
- 600 会话成本: ~$0.60 (Vertex AI batch)

Grok (.json) ---------+ (24 个精选谓词) |
Warp (SQLite) --------+ v
 Apache Jena Fuseki
 |
 v

**实体链接**(上下文感知,Agentic):
1. 通过 entity_aliases.json 标准化(161 个映射)
2. 检查 SQLite 缓存
3. 未命中 → LangGraph ReAct agent (LLM + Wikidata API)
4. 置信度阈值 0.7 → owl:sameAs 链接

## 24 个精选谓词
封闭世界设计,LLM 只能用这些谓词:

| 类别 | 谓词 |
| 依赖 | uses, dependsOn, requires, builtWith |
| 能力 | enables, provides, solves, produces |
| 结构 | isPartOf, hasPart, extends, implements |
| 分类 | isTypeOf, broader, narrower |
| 基础设施 | deployedOn, storesIn, queriedWith, configures |
| 关系 | integratesWith, composesWith, alternativeTo, servesAs, relatedTo |

## 快速开始
```bash

# 1. 克隆
git clone https://github.com/robertoshimizu/session-graph.git
cd session-graph

# 2. 配置
cp .env.example .env

# 3. 安装
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# 4. 启动 Fuseki
docker run -d --name fuseki -p 3030:3030 stain/jena-fuseki

# 5. 处理单个会话
python -m pipeline.jsonl_to_rdf path/to/session.jsonl output/session.ttl

# 6. 链接实体到 Wikidata
python -m pipeline.link_entities --input output/*.ttl --output output/wikidata_links.ttl

# 7. 加载到 Fuseki
python -m pipeline.load_fuseki output/*.ttl

# 8. 查询 http://localhost:3030

## 核心洞察
> "A knowledge graph without relationships is just a tag cloud. The minimum viable extraction unit is (subject, predicate, object), not [topic1, topic2, topic3]."

没有关系的知识图谱只是标签云.最小可行的提取单元是 (主语, 谓词, 宾语),不是 [主题1, 主题2, 主题3].

**一句话**:Session-graph 让你的 AI 编码会话不再是一次性的,而是可查询,可关联,可扩展的知识资产.
