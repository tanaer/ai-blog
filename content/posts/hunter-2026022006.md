YARA 是恶意软件检测的行业标准.但你如果用 Go 写过 YARA 相关项目,就知道 CGo 有多痛苦--需要 C 编译器,pkg-config,预装的 libyara,交叉编译简直是噩梦.

刚刚在 HackerNews 上看到的:**Sansec 用纯 Go 重写了 YARA 引擎,叫 YARGo,针对源代码扫描做了深度优化,速度提升 6.8 倍.**

## 他们为什么要重写?
Sansec 用 YARA 做 eComscan 和威胁监控,每天扫描几十万电商站点.但原版 YARA 有几个痛点:

1. **CGo 太麻烦**:需要专门的构建服务器来兼容老系统
2. **无法静态编译**:丢掉了 Go 最大的优势之一
3. **为二进制设计**:YARA 内部优化是针对恶意软件分析的,不是源代码扫描

他们主要扫描 PHP,JavaScript,HTML 这类文本文件,所以干脆自己写了一个.

## YARGo 做了什么优化?

### 1. 完整字符串放进 Aho-Corasick 自动机
YARA 会把每个 atom 截断到 4 字节.一个 19 字节的模式 `eval(base64_decode(` 只剩 4 字节,误匹配率很高.

YARGo 把完整字符串字面量放进 AC 自动机.19 字节就是 19 字节,只有真正匹配时才触发.代价是自动机大一点,但内存消耗只是 MB 级别.

### 2. 更聪明的 regex atom
YARA 可以用 1 字节的 regex atom.这会导致大量无意义的验证.

YARGo 要求 atom 最少 3 字节.256³ = 1670 万种可能,误匹配概率大幅下降.

### 3. 针对源代码的评分函数
YARA 的 atom 评分是通用的:`0x00`,`0x20` 这些常见字节会被惩罚.

YARGo 专门针对 Web 源代码调优:
- PHP/JS 常见 token(return,function,var,`();`)直接禁止, 字母字节(源代码里很常见)评分低,非 ASCII 字节评分高, 挑选在实际扫描文件中区分度高的 atom

### 4. 真实数据训练的字节频率
AC 预过滤用的字节频率数据来自真实电商站点,扫描器知道哪些字节真正稀有,可以更高效地跳过.

## 性能数据
eComscan 每天跑 57,000 次扫描.YARGo 已经在生产环境跑了两周:

平均扫描时间, 旧引擎=12.5 分钟, YARGo=< 2 分钟
中位扫描时间, 旧引擎=-, YARGo=< 1 分钟
速度提升, 旧引擎=-, YARGo=**6.8x**
两周节省 CPU, 旧引擎=-, YARGo=**116,000 小时**

## 怎么用?
```bash

# 安装
go get github.com/sansecio/yargo

# 使用(与 go-yara API 兼容)
import "github.com/sansecio/yargo"

compiler, _ := yargo.NewCompiler()
compiler.AddFile("rules.yar", "")
scanner, _ := yargo.NewScanner(compiler.GetRules())
matches, _ := scanner.ScanFile("target.php", 0, 0)
```

## 实际应用场景
**场景一:Webshell 检测**

你有大量 PHP/JS 文件需要检测后门,传统 YARA 对源代码扫描不够快.YARGo 针对这类场景优化,12 分钟的扫描压缩到 2 分钟以内.

**场景二:CI/CD 集成**

想在每个 PR 里跑 YARA 规则检查恶意代码?YARGo 是纯 Go,可以打包成静态二进制,扔进 Docker 镜像,无需安装 C 依赖.

**一句话**:如果你用 Go 做 Web 安全扫描,YARGo 值得一试.MIT 开源,GitHub: [github.com/sansecio/yargo](https://github.com/sansecio/yargo)
